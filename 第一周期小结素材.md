### 2. 过程中有没有遇到不顺或卡住的情况？后来是怎么破局的？

当前主要卡点：

- 飞书官方 API 暂未发放，无法直接实现真实 IM、Docs 和 Slides 操作。
- 比赛题目要求较宽，容易做成“什么都想做”的大而散系统。
- Agent 输出如果直接用于文档和 PPT，可能出现格式不稳定、结构不一致的问题。
- 多端协同和办公套件自动化都比较复杂，如果一开始就做完整能力，开发风险较高。

破局方式：

- 采用 Adapter-first 技术路线，把飞书 API、larksuite CLI 和 Mock 实现都封装在 Tool Adapter 后面。
- 先确定可演示的最短闭环：IM 指令 -> Agent 规划 -> 文档生成 -> PPT 生成 -> 总结交付。
- 将任务拆成 A-F 六个独立场景，保证每个场景都可以单独演示，组合起来又能形成完整链路。
- 对豆包 2.0 Pro 的输出先设计结构化 JSON 约束，后续通过 JSON Schema、重试和 Verifier 提升稳定性。
- 多端协同第一阶段只同步任务状态和产物元信息，暂不直接挑战复杂文档级冲突合并。

后续可以直接复用的 Planner Prompt 草案：

```text
你是一个办公协同 AI Agent 的任务规划器。
用户会给出一个来自 IM 对话的自然语言指令。
请你把指令拆解为可执行任务计划。

要求：
1. 必须输出 JSON。
2. 不要输出 Markdown。
3. steps 中的每个步骤都必须包含 title、tool、inputSummary、expectedOutput。
4. 如果信息不足，必须在 requiredConfirmations 中提出澄清问题。
5. 如果任务涉及文档、PPT 或归档，需要明确对应产物类型。

输出字段：
- goal
- steps
- requiredConfirmations
- risks
```

# 第四周期任务规划

周期建议：2026.05.02 - 2026.05.05

主题：从“可演示闭环”升级为“高可信、可复盘、可迁移”的 Agent-Pilot 作品。

## 一、周期目标

第四周期不再把精力放在是否购买服务器这一单点上，而是围绕比赛评分点继续强化三件事：

1. **Agent 主驾驶感**：让任务规划、工具调用、确认、失败恢复和交付摘要更像一个可解释的 Agent，而不是固定脚本。
2. **Office 产物质量**：让 Docs / Slides 产物更接近真实汇报材料，减少“空白 PPT / 结构单薄 / 演示痕迹重”的风险。
3. **演示稳定性**：在不依赖官方服务器的前提下，用低成本方案保证飞书入口、仪表盘、长连接和本地 Agent 可被快速检查、恢复和替换。

## 二、P0 必须完成

| 编号 | 任务 | 验收标准 | 价值 |
| --- | --- | --- | --- |
| P0-1 | 固化低成本部署路线 | 项目内有部署矩阵、人工操作清单、保底方案和迁移方案 | 不再被服务器资源卡住 |
| P0-2 | 增强交付前诊断 | 一条命令生成 typecheck/build/e2e/runtime 结果和人工待办 | 比赛前快速自检 |
| P0-3 | 飞书 IM 端到端稳定性复测 | 群内 `/agent`、确认、进度、取消、回发链接全部跑通 | 覆盖 IM 主入口 |
| P0-4 | Slides 内容和飞书渲染复核 | 飞书打开 PPT 非空，至少 5 页结构完整 | 避免核心产物失分 |
| P0-5 | 任务失败恢复策略 | 失败步骤可在事件里定位，并能给用户明确下一步 | 增强工程可信度 |

| P1-5 | 形成技术亮点页 | 明确 Agent-Pilot、模块化场景、低成本部署、多端一致性 |

## 五、开发范式执行要求

第四周期继续按“探索 - 计划 - 实现 - 验证 - 复盘”推进：

1. 开发前先明确验收标准，不为了炫技破坏主链路。
2. 每次改动尽量小步提交，保证 TypeScript 类型、E2E 和运行诊断可过。
3. Agent 能力优先服务赛题要求：IM、Doc、Slides、自然语言、多端协同。
4. 任何云服务、域名、服务器、飞书后台发布都记录为人工操作项。
5. 文档和代码同步演进，避免“代码能跑但说明过期”。

# 低成本部署方案与人工操作清单

更新时间：2026.05.01

## 1. 结论

官方暂不提供服务器时，本项目不是只能走“购买服务器 + 购买域名”这一条路。当前可以保留三档方案：

| 档位 | 方案 | 成本 | 是否需要域名 | 是否适合比赛演示 | 结论 |
| --- | --- | --- | --- | --- | --- |
| A | Cloudflare Pages + 本地长连接 + Quick Tunnel | 近似 0 元 | 否 | 适合现场演示 | 当前保底方案，已可用 |
| B | Cloudflare Pages 边缘代理 / Worker Relay + Quick Tunnel | 近似 0 元 | 否 | 适合短期稳定入口 | 飞书后台少改 URL，但本机仍需运行 |
| C | Named Tunnel + 自有域名 + 本机/宿舍电脑常驻 | 域名成本 | 是 | 适合答辩前稳定化 | API URL 稳定，仍依赖本机在线 |
| D | 低价 VPS / 轻量云服务器 + Cloudflare Tunnel | 服务器成本 | 可选 | 最适合远程长期体验 | 稳定性最好，运维工作最多 |
| E | 全云 Serverless Worker / Durable Object | 低成本但改造大 | 否或可选 | 第四周期探索项 | 架构漂亮，但会影响主链路风险 |

当前推荐排序已经调整为：

1. 当前主线：轻量云服务器 + Nginx + HTTPS + systemd 常驻，入口为 `https://agent-pilot.47-236-122-49.sslip.io`。
2. 保底回退：本地 API + lark-cli 长连接 + Quick Tunnel/Pages 代理。
3. 后续增强：购买正式域名，把 `A` 记录指向服务器公网 IP。
4. 不建议现在大改成纯 Worker：当前 Agent 还依赖 `lark-cli`、本地状态、Office CLI 操作和长连接，贸然迁移会破坏已经跑通的主链路。

## 2. 为什么不能完全依赖飞书开放平台托管

飞书开放平台负责应用能力、机器人、事件订阅、权限、菜单和入口展示，但不会自动运行本仓库里的 Node API、React 仪表盘、豆包调用和本地 `lark-cli`。因此至少需要一个“运行时”承载：

- Agent API。
- 飞书长连接事件桥接。
- 豆包 2.0 Pro 调用。
- 飞书 Docs / Slides / IM 回发操作。
- 任务状态、幂等记录和演示日志。

所以“部署”不是额外负担，而是比赛题目中多端协同和工程落地能力的一部分。

## 3. 低成本可选路线

### 3.1 保底路线：Cloudflare Pages + Quick Tunnel

当前已经支持：

```text
飞书机器人菜单 / 网页应用
  -> https://feishu-ai-im-collab-assistant.pages.dev
  -> Pages 边缘代理
  -> Quick Tunnel
  -> 本地 Agent API
  -> lark-cli 长连接 / Docs / Slides / IM
```

优点：

- 不买域名。
- 不买服务器。
- 改动最小。
- 适合现场演示和迭代开发。

风险：

- 电脑必须在线。
- Quick Tunnel URL 可能变化。
- 网络代理、校园网、电脑休眠都会影响演示。

适用命令：

```powershell
npm run demo:stack
npm run deploy:web-proxy:cloudflare
npm run doctor:runtime
```

### 3.2 免费/轻量 Serverless 路线

可用来承载前端、边缘代理、轻量 webhook、健康检查和展示页，但当前不适合直接承载完整 Agent 执行器。

可选平台：

| 平台 | 适合放什么 | 不适合放什么 |
| --- | --- | --- |
| Cloudflare Pages | 仪表盘、静态资源、边缘代理 | 直接运行 lark-cli 长连接 |
| Cloudflare Workers | webhook relay、健康检查、轻量 API、状态路由 | 长时间执行 Office CLI |
| Vercel / Netlify | 前端和轻量函数 | 长连接、CLI 常驻进程 |
| Railway / Render | Node 服务原型 | 成本和休眠策略要提前验证 |

推荐使用方式：

- 前端继续在 Cloudflare Pages。
- Agent API 暂时仍运行在本地或 VPS。
- Worker 只做 relay、边缘健康检查和轻量防护。

### 3.3 低价 VPS 路线

如果后续决定买服务器，推荐配置：

```text
2 vCPU / 2-4 GB RAM / 40 GB SSD
Ubuntu 22.04 或 24.04
Node.js 20+
pm2 或 systemd 常驻 API 和 lark-event-bridge
Cloudflare Tunnel 或 Nginx + HTTPS
```

优点：

- 评委远程体验更稳。
- 不依赖本机网络和电源。
- 更符合“可用工程系统”的观感。

风险：

- 需要迁移 `.env`、`lark-cli` 登录态和本地状态。
- 需要做好密钥管理。
- 需要留意服务器地区访问飞书、火山方舟的网络质量。

### 3.4 自有域名是不是必须

不是必须。

不买域名也可以：

- 飞书菜单填 `https://feishu-ai-im-collab-assistant.pages.dev`。
- Pages 通过边缘代理转发到当前 API。
- Worker Relay 可以使用 `workers.dev` 子域名。

需要域名的场景：

- 想要一个稳定、正式、好看的 API URL，例如 `https://api.agent-pilot.example.com`。
- 想使用 Named Tunnel 的 public hostname。
- 想让评委长期访问更可信。

## 4. 当前必须人工操作的事项

| 事项 | 谁做 | 原因 | 当前建议 |
| --- | --- | --- | --- |
| 决定是否购买 VPS | 你 | 需要账号、支付和实名 | 先不买，等主链路再压测一轮 |
| 若购买 VPS，完成账号实名和付款 | 你 | 涉及个人信息与支付 | 只买短期低配即可 |
| 若购买域名，完成域名注册和 DNS 接入 | 你 | 涉及实名/支付 | 非必须，可延后 |
| 飞书后台创建版本并发布 | 你 | 需要网页会话与应用管理员身份 | 菜单改完后必须发布版本 |
| 飞书客户端验证机器人菜单缓存 | 你 | 客户端可能有 5 分钟缓存 | 发布后重新打开会话检查 |
| lark-cli 登录态迁移到服务器 | 你 + 我辅助 | 需要交互授权 | 若买 VPS 再做 |

## 6. 第四周期部署策略

第四周期已经采用轻量服务器作为主线，Cloudflare / 本地方案作为回退：

```text
主轨：轻量云服务器 + 免费 sslip.io 域名 + HTTPS + systemd 常驻。
回退：本地长连接 + Cloudflare Pages / Quick Tunnel，保证现场仍可救场。
```

这样的好处是：正常答辩时飞书桌面端和移动端都能稳定打开同一个 HTTPS 地址；如果服务器临时异常，仍能回退到本地演示路线。

# Agent-Pilot 项目技术方案

## 1. 项目定位

本项目面向飞书比赛题目“基于 IM 的办公协同智能助手”，目标是实现一个以 AI Agent 为主驾驶、GUI 为辅助仪表盘的多端协同办公助手。

用户通过 IM 对话中的自然语言指令发起任务，Agent 负责理解意图、规划步骤、调用办公套件能力，并最终完成从需求捕捉、文档沉淀、演示稿生成到汇报归档的闭环。

核心原则：

- AI Agent 是 Pilot，负责理解、拆解、执行和校验。
- GUI 是 Co-pilot，负责展示进度、承接确认、支持人工微调。
- 各场景模块可以独立演示，也可以按任务自由组合编排。
- 当前优先使用飞书开放平台应用 + `larksuite/cli` + 长连接事件桥接落地真实 Demo，同时保留 Adapter 边界，后续可迁移到飞书 OpenAPI SDK 或云端 Worker 架构。

## 2. 目标场景

### 场景 A：IM 意图入口

用户在飞书群聊或单聊中通过文本或语音输入任务，例如：

> 帮我把刚才关于校园活动报名系统的讨论整理成需求文档，并生成一份 5 页汇报 PPT。

系统需要完成：

- 捕捉自然语言意图。
- 识别上下文来源，例如当前会话、历史消息、附件、用户补充说明。
- 创建 Agent 任务并进入规划阶段。

### 场景 B：任务理解与规划

Agent 使用豆包 2.0 Pro 对用户意图进行结构化拆解，生成可执行计划。

计划应包含：

- 任务目标。
- 需要读取的上下文。
- 需要生成或修改的产物。
- 子任务顺序。
- 需要用户确认的关键节点。
- 失败重试或降级策略。

### 场景 C：文档 / 白板生成与编辑

Agent 根据 IM 讨论内容生成结构化文档，例如需求文档、会议纪要、方案说明或项目 Brief。

支持能力：

- 自动提取背景、目标、用户角色、功能列表、风险和待办事项。
- 根据用户自然语言进行增删改。
- 记录文档版本和 Agent 修改原因。
- 后续接入飞书 Docs 或自由画布 API。

### 场景 D：演示稿 / 自由画布生成与排练

Agent 将文档内容转化为演示材料。

支持能力：

- 从文档生成 PPT 大纲。
- 生成页面标题、要点、讲稿和备注。
- 支持用户按页修改，例如“第三页改成架构图风格”。
- 支持演练反馈，例如指出逻辑薄弱页、压缩汇报时长。

### 场景 E：多端协作与一致性

移动端和桌面端共享同一任务状态、产物状态和用户确认状态。

第一阶段重点同步：

- 当前任务进度。
- Agent 正在执行的步骤。
- 文档和演示稿的产物元信息。
- 用户确认或驳回动作。

后续增强：

- 离线编辑。
- 冲突检测。
- 基于版本或 CRDT 的内容合并。

### 场景 F：总结与交付

Agent 在任务结束后输出适合汇报和归档的交付内容。

包括：

- 最终文档链接。
- 最终演示稿链接或导出文件。
- 任务总结。
- 关键决策记录。
- 后续待办事项。

## 3. 总体架构

```text
Mobile Client / Desktop Client
              |
       Realtime Sync Layer
              |
        Agent Orchestrator
              |
 Planner -> Executor -> Verifier -> Memory
              |
        Tool Adapter Layer
              |
 IM Adapter / Docs Adapter / Slides Adapter / Canvas Adapter / Export Adapter
              |
 Feishu Open Platform / larksuite CLI / Doubao 2.0 Pro / Cloudflare Pages
```

## 4. 核心模块

### 4.1 Client Layer

负责移动端与桌面端交互体验。

职责：

- 展示任务列表和任务详情。
- 展示 Agent 执行进度。
- 提供必要确认按钮和自然语言输入框。
- 显示生成中的文档、PPT 或画布结果。

建议技术路线：

- 桌面端：React + Electron 或 Tauri。
- 移动端：Expo React Native，或先以移动端 Web / PWA 验证。

### 4.2 Realtime Sync Layer

负责多端状态同步。

第一阶段建议使用 WebSocket 同步任务状态，不直接处理复杂文档协同。

核心事件：

- `task.created`
- `task.planned`
- `step.started`
- `step.completed`
- `artifact.created`
- `artifact.updated`
- `artifact.verified`
- `user.confirmed`
- `user.rejected`
- `task.delivered`

### 4.3 Agent Orchestrator

Agent 主控模块，负责把用户意图转化为可执行工作流。

内部组件：

- Planner：任务规划。
- Executor：调用工具执行。
- Verifier：检查产物是否满足目标。
- Memory：保存会话上下文、任务状态、产物版本。

当前已落地：

- `Planner`：通过豆包 / mock LLM 生成执行计划。
- `ContentComposer`：生成需求文档、Slides 大纲和讲稿。
- `ArtifactVerifier`：对 Docs、Slides、交付摘要做轻量验收。
- `TaskStore`：持久化 Task、Event 和 Artifact，支持 API 重启后恢复。
- `HandledMessageStore`：持久化飞书 messageId，避免重复执行。

### 4.4 Tool Adapter Layer

为了应对飞书 API 暂未发放的问题，所有办公套件能力先通过 Adapter 抽象。

示例接口：

```ts
export interface OfficeToolAdapter {
  readMessages(input: ReadMessagesInput): Promise<MessageContext>;
  sendMessage(input: SendMessageInput): Promise<void>;
  createDoc(input: CreateDocInput): Promise<Artifact>;
  updateDoc(input: UpdateDocInput): Promise<Artifact>;
  createSlides(input: CreateSlidesInput): Promise<Artifact>;
  exportArtifact(input: ExportArtifactInput): Promise<ExportResult>;
}
```

后续可以提供多种实现：

- `MockOfficeToolAdapter`：比赛早期本地演示使用。
- `FeishuApiAdapter`：接入飞书开放平台 API。
- `LarkCliAdapter`：基于官方 `larksuite/cli` 调用 IM、Docs、Slides 等能力。

当前 `LarkCliAdapter` 已增加命令超时和只读重试。创建 Docs / Slides / 群回发不自动重试，避免真实飞书里重复创建产物或重复发消息。

## 5. 数据模型草案

### Task

```ts
type Task = {
  id: string;
  title: string;
  source: "im" | "desktop" | "mobile";
  status: "created" | "planning" | "running" | "waiting_user" | "completed" | "cancelled" | "failed";
  userIntent: string;
  plan?: AgentPlan;
  artifacts: Artifact[];
  createdAt: string;
  updatedAt: string;
};
```

### AgentPlan

```ts
type AgentPlan = {
  goal: string;
  steps: AgentStep[];
  requiredConfirmations: string[];
  risks: string[];
};
```

### AgentStep

```ts
type AgentStep = {
  id: string;
  title: string;
  tool: string;
  status: "pending" | "running" | "completed" | "failed" | "skipped";
  inputSummary: string;
  outputSummary?: string;
};
```

### Artifact

```ts
type Artifact = {
  id: string;
  type: "doc" | "slides" | "canvas" | "summary" | "export";
  title: string;
  url?: string;
  version: number;
  createdBy: "agent" | "user";
  updatedAt: string;
};
```

## 6. 豆包 2.0 Pro 使用方式

豆包 2.0 Pro 在项目中承担核心 Agent 推理能力。

主要使用场景：

- 意图识别。
- 任务规划。
- IM 讨论总结。
- 文档生成。
- PPT 大纲生成。
- 讲稿生成。
- 产物质量检查。
- 主动澄清问题生成。

为了保证稳定性，建议所有关键 Agent 输出采用结构化 JSON。

示例要求：

```text
你是办公协同 Agent 的 Planner。
请把用户意图拆解成可执行步骤。
必须输出 JSON，不要输出 Markdown。
字段包括 goal、steps、requiredConfirmations、risks。
```

## 7. 演示优先级

第一优先级：

- IM 自然语言触发。
- Agent 规划步骤可视化。
- 自动生成文档。
- 从文档生成演示稿。
- 移动端与桌面端状态同步。

第二优先级：

- 用户自然语言修改文档或 PPT。
- Agent 主动澄清任务。
- 生成汇报讲稿。
- 导出和归档。

第三优先级：

- 离线编辑与恢复同步。
- 富媒体插入。
- 复杂冲突解决。

## 8. 当前风险

| 风险 | 影响 | 解决思路 |
| --- | --- | --- |
| 飞书 API 暂未发放 | 无法直接接真实 IM、Docs、Slides | 先做 Adapter 和 Mock，实现可替换架构 |
| LLM 输出不稳定 | 任务规划和文档格式可能漂移 | 使用 JSON Schema、重试、Verifier 校验 |
| 多端同步复杂 | 状态和内容可能不一致 | 第一阶段只同步任务状态和产物元信息 |
| PPT API 能力边界未知 | 生成效果可能受限 | 同时保留自由画布出口 |
| Demo 链路过长 | 容易演示失败 | 主链路保持短闭环，其他能力单独展示 |
| API 入口不稳定 | 评委远程打不开或 Quick Tunnel 变化 | 当前用 Pages 边缘代理过渡，推荐 Named Tunnel + 自有域名或 2C4G 云服务器 |

## 9. 下一步

- 在飞书开放平台发布新版机器人菜单，补“打开仪表盘”入口。
- 若有自有域名，把 Named Tunnel `feishu-agent-api` 绑定到 `api.your-domain.com`。
- 如果需要远程长期体验，准备 2 vCPU / 4 GB RAM 云服务器并迁移 API 与长连接桥接。
- 继续增强 Slides XML Builder 的版式、讲者备注和流程图页。
- 将 Web 仪表盘扩展为飞书内嵌页面、移动端视图、桌面端视图和演示模式。

## 10. 当前工程骨架

已落地第一版 monorepo 工程：

```text
apps/api
  Agent 后端、任务编排、豆包 Provider、Office Adapter、WebSocket 同步

apps/web
  React + Vite 多端协同仪表盘

packages/shared
  Task、AgentPlan、AgentStep、Artifact、AgentEvent 等共享类型

tests/e2e
  Playwright 浏览器烟测
```

当前已跑通的最小闭环：

```text
移动端 IM 输入
  -> API 创建任务
  -> Agent Planner 生成步骤
  -> Mock IM 读取上下文
  -> Mock Doc 生成需求文档
  -> Mock Slides 生成 5 页 PPT 内容
  -> 生成 3 分钟讲稿
  -> 生成交付摘要
  -> WebSocket 同步到桌面仪表盘
```

当前真实飞书闭环正在推进为：

```text
飞书群内 /agent 或 @机器人
  -> lark-cli 长连接接收事件
  -> 安全过滤、幂等和同群会话互斥
  -> Agent 规划并请求确认
  -> 用户回复确认 / 进度 / 取消
  -> 创建真实 Docs + Slides + 讲稿
  -> ArtifactVerifier 校验产物
  -> 群内回发交付链接
  -> Cloudflare Pages 仪表盘实时展示状态
```

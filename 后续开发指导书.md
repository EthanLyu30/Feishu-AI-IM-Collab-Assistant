当前仍处于工程早期，真实飞书能力已经完成 IM 读取 + Docs + Slides + 群回发的小闭环雏形。下一步重点是群内自动触发、飞书应用内嵌页面、Slides 版式增强和错误处理。

## 2. 当前能否操作飞书

可以，但建议按“低风险到高风险”的顺序接入。

当前 `lark-cli` 已经可用：

```powershell
lark-cli doctor
lark-cli auth status
```

已确认可用能力：

- IM：读取群消息、搜索消息、发送交付摘要。
- Docs：创建文档、读取文档、更新文档、插入媒体。
- Slides：创建演示稿、更新演示页。
- Whiteboard：创建和更新白板节点。
- Drive：上传和下载文件。

但执行真实写操作前，需要明确目标：

- 要操作哪个群聊，需要 `chat_id`。
- 要创建文档到哪个文件夹，需要 `folder_token`，不填则通常创建到默认位置。
- 要更新哪个文档，需要文档 URL 或 token。
- 要发送消息到谁，需要 `chat_id` 或用户 `open_id`。

## 3. lark-cli 常用命令

### 3.1 检查登录状态

```powershell
lark-cli auth status
lark-cli doctor
```

### 3.2 搜索群聊

```powershell
lark-cli im +chat-search --query "群聊关键词" --format pretty
```

拿到 `chat_id` 后，可以读取消息。

### 3.3 读取群聊消息

```powershell
lark-cli im +chat-messages-list --chat-id "oc_xxx" --page-size 20 --sort desc --format pretty
```

可加时间范围：

```powershell
lark-cli im +chat-messages-list `
  --chat-id "oc_xxx" `
  --start "2026-04-24T09:00:00+08:00" `
  --end "2026-04-24T10:00:00+08:00" `
  --sort asc `
  --format pretty
```

### 3.4 发送 IM 消息

首次发送前，如果提示缺少 scope，先执行：

```powershell
lark-cli auth login --scope "im:message.send_as_user"
```

完成浏览器授权后再发送：

```powershell
lark-cli im +messages-send --as user --chat-id "oc_xxx" --text "Agent-Pilot 已完成文档生成。"
```

Markdown 消息：

```powershell
lark-cli im +messages-send --as user --chat-id "oc_xxx" --markdown "## 任务完成`n- 需求文档已生成`n- PPT 已生成"
```

### 3.5 创建飞书文档

```powershell
lark-cli docs +create `
  --as user `
  --title "校园活动报名系统需求文档" `
  --markdown "@./outputs/requirements.md"
```

如果要创建到指定文件夹：

```powershell
lark-cli docs +create `
  --as user `
  --folder-token "fldcn_xxx" `
  --title "校园活动报名系统需求文档" `
  --markdown "@./outputs/requirements.md"
```

### 3.6 更新飞书文档

追加内容：

```powershell
lark-cli docs +update `
  --as user `
  --doc "https://xxx.feishu.cn/docx/xxx" `
  --mode append `
  --markdown "## 权限管理补充`n学生和老师的权限边界如下..."
```

覆盖全文：

```powershell
lark-cli docs +update `
  --as user `
  --doc "https://xxx.feishu.cn/docx/xxx" `
  --mode overwrite `
  --markdown "@./outputs/requirements.md"
```

### 3.7 创建飞书演示稿

`slides +create` 需要 slide XML 字符串数组，不是普通 Markdown。当前代码已经做了一个简版 `Markdown -> Slides XML` 转换，可生成真实可打开的飞书 Slides。

示例：

```powershell
lark-cli slides +create `
  --as user `
  --title "校园活动报名系统汇报 PPT" `
  --slides '["<slide><text>校园活动报名系统</text></slide>"]'
```

后续需要把转换器抽到独立模块，并增强版式、标题层级、讲者备注和流程图页面。

## 4. 代码接入顺序

建议按以下顺序推进，不要一上来就把 IM、Docs、Slides 全部真实接入。

### Step 1：接入豆包 2.0 Pro

目标：让 Planner 和内容生成真正使用豆包，而不是 Mock。

本地新建 `.env`：

```env
AGENT_LLM_MODE=doubao
ARK_BASE_URL=https://ark.cn-beijing.volces.com/api/v3
ARK_ENDPOINT_ID=你的 EP ID
ARK_API_KEY=你的 Ark API Key
OFFICE_ADAPTER=mock
LARK_DEFAULT_CHAT_ID=
```

验证：

```powershell
npm run dev
```

打开：

```text
http://localhost:5173
```

点击发送 IM 指令，检查：

- Planner 是否由豆包生成。
- 文档是否由豆包生成。
- PPT 大纲是否由豆包生成。
- 追加修改是否由豆包生成。

### Step 2：完善豆包输出稳定性

需要补的能力：

- Planner JSON Schema 校验。
- JSON 解析失败重试。
- LLM 输出超时处理。
- 生成结果质量检查。
- 失败后回退到 Mock 模板。

建议新增：

```text
apps/api/src/agent/schemas.ts
apps/api/src/agent/Verifier.ts
```

### Step 3：接入真实 Docs 创建

目标：先让需求文档真的创建到飞书。

修改：

```text
apps/api/src/adapters/LarkCliAdapter.ts
```

当前 `createDoc` 已经通过临时 Markdown 文件接到：

```powershell
lark-cli docs +create --as user --title ... --markdown ...
```

注意：

- 不要把 Markdown 内容直接塞到命令参数里，长文本容易转义失败。
- 更稳妥做法是写入临时文件，再用 `--markdown @file`。当前代码已采用这个方式。
- 临时文件应放到 `.tmp/`，并加入 `.gitignore`。
- 当前已补充对 CLI JSON / pretty 输出的稳定解析，把真实文档链接写回 Artifact。
- `updateDoc` 已接入 `docs +update --mode overwrite`，自然语言追加修改可以更新真实文档。

### Step 4：接入真实 Slides 创建

目标：让 PPT 真的生成到飞书 Slides。

当前已有内联简版转换器：

```text
Markdown PPT 大纲 -> Slides XML array
```

后续建议抽成独立文件：

```text
apps/api/src/slides/SlidesXmlBuilder.ts
```

第一版只支持：

- 标题页。
- 要点页。
- 流程页。
- 总结页。

先追求稳定生成，不追求复杂视觉。

### Step 5：接入真实 IM 读取和发送

目标：从飞书群聊读取消息，并把 Agent 结果发回群聊。

需要先解决：

- 如何配置 `chat_id`。
- 是从固定群读取，还是从用户选择的群读取。
- 是否需要监听事件回调，还是先由用户在 Web 中输入 `chat_id` 手动触发。

第一版建议：

```text
用户在 Web 输入 chat_id
  -> Agent 读取最近 20 条消息
  -> 生成文档和 PPT
  -> 把交付摘要发回群聊
```

第二版再做事件订阅和机器人自动响应。

### Step 6：多端协同增强

当前 Web 已经模拟移动端和桌面端。

注意：当前 Web 仪表盘不是最终产物形态，而是开发期的控制台 / 演示仪表盘。最终更适合包装成飞书应用内页面：群聊里由 IM 指令触发，飞书应用页面展示进度、产物、确认节点和错误信息。

下一步：

- 把左侧 IM 面板抽象为移动端视图。
- 把右侧仪表盘抽象为桌面端视图。
- 增加任务历史。
- 增加任务详情页。
- 增加产物打开按钮。
- 增加“需要用户确认”的暂停节点。

## 6. 后续演示路线

### 第一版真实飞书 Demo

```text
Web 输入 chat_id
  -> 读取飞书群聊最近消息
  -> 豆包生成任务计划
  -> 创建飞书需求文档
  -> 创建飞书 Slides
  -> 飞书群聊发送交付摘要
```

### 第二版 Agent 主驾驶 Demo

```text
飞书 IM 中 @Agent
  -> Agent 监听事件
  -> 自动读取上下文
  -> 生成计划并请求确认
  -> 用户回复“确认”
  -> Agent 生成 Doc + Slides
  -> 回发链接和讲稿
```

### 第三版多端协同 Demo

```text
移动端飞书触发
  -> 桌面端 Web 仪表盘同步进度
  -> 桌面端确认/修改
  -> 移动端收到最终交付
```

## 7. 风险与处理

| 风险 | 处理 |
| --- | --- |
| 豆包输出格式不稳定 | JSON Schema + 重试 + Verifier |
| Slides XML 生成失败 | 先限制模板类型，逐步扩展 |
| IM 消息格式复杂 | 第一版只处理文本，图片/文件后续做 |
| 长 Markdown 命令转义失败 | 使用临时文件 + `--markdown @file` |
| token 过期 | 用 `lark-cli auth status` 检查，必要时重新 login |
| 真实群聊写入风险 | 开发阶段使用测试群 |

## 8. 当前最优先任务

建议接下来只做三件事：

1. 接入飞书事件订阅 / 机器人回调，让群内消息直接触发 Agent。
2. 将 Web 仪表盘包装成飞书应用内页面。
3. 把 Slides XML Builder 独立出来并增强版式。

这三件事完成后，项目就从 Mock Demo 进入真实飞书生态 Demo。

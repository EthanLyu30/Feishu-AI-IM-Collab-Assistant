# Agent-Pilot Development Guidelines

本项目后续开发遵循 Karpathy-inspired coding workflow，并结合飞书比赛项目的实际约束执行。

参考来源：

- Karpathy-inspired coding workflow: `forrestchang/andrej-karpathy-skills`
- DESIGN.md workflow: `VoltAgent/awesome-design-md`

## 1. 编码前思考

不要静默假设。

开始非琐碎任务前，先写清楚：

- 当前目标是什么。
- 已知事实是什么。
- 不确定点是什么。
- 是否存在更简单方案。
- 本轮成功标准是什么。

如果任务依赖飞书真实资源，例如 `chat_id`、文档 token、Slides token、文件夹 token，必须先确认资源来源，不能在代码中硬编码真实账号或真实群聊。

开始开发一个新模块前，先做一次轻量参考检索：

- 优先查 GitHub 上 star 数高、维护活跃、和当前任务相关的 skill / guideline / example 项目。
- 只借鉴工作流、设计规范和验证方法，不盲目搬运复杂架构。
- 如果参考项目和当前需求不匹配，明确说明不采用的原因。

## 2. 简洁优先

优先做最小可验证闭环。

本项目的默认推进顺序：

```text
Mock 可跑
  -> 单个真实能力可跑
  -> 组合链路可跑
  -> UI 展示和体验优化
  -> 多端和离线增强
```

禁止为了“以后可能有用”提前做大抽象。新的抽象必须满足至少一个条件：

- 消除真实重复逻辑。
- 明显降低 Agent 编排复杂度。
- 是飞书 API / CLI 接入边界所必需。

## 3. 精准修改

每一行改动都要能追溯到本轮任务。

开发时遵守：

- 不顺手重构无关文件。
- 不改无关格式。
- 不删除不是本轮改动造成的代码。
- 不把 `.env`、App Secret、Ark API Key、token、CLI 日志提交到仓库。
- 真实飞书写操作只在测试群、测试文档或明确授权资源中进行。

如果发现无关问题，记录到后续任务，不在当前改动里顺手修。

## 4. 目标驱动执行

每个任务必须有可验证成功标准。

示例：

```text
目标：接入真实飞书 Docs 创建
成功标准：
1. 使用 Mock 模式时 E2E 仍通过。
2. 使用 lark-cli 模式时能创建真实飞书文档。
3. Artifact 中能拿到可打开的文档链接。
验证：
- npm run typecheck
- npm run test:e2e
- 手动执行一次测试群 / 测试文档创建
```

禁止用“应该可以”“大概完成”作为结束标准。

## 5. 设计范式

UI 开发遵循 DESIGN.md 思路：先定义视觉和交互规则，再实现界面。

开发新的页面、仪表盘、移动端视图或桌面端视图前，先补充或引用对应设计约束：

- 视觉主题与产品气质。
- 颜色语义和使用范围。
- 字体层级和信息密度。
- 组件规则：按钮、卡片、输入框、状态标签、任务列表、产物卡片。
- 布局规则：移动端 / 桌面端如何切换，哪些区域必须保持同步可见。
- Do / Don't：明确哪些设计不能做。
- 响应式行为和触控目标。

本项目默认设计方向：

```text
工作流仪表盘，不做营销页。
AI Agent 是主驾驶，GUI 是任务仪表盘和确认台。
界面应克制、清晰、可扫描，适合真实办公协作场景。
优先展示任务状态、Agent 步骤、产物链接、用户确认和错误信息。
```

如果要借鉴外部 DESIGN.md，优先参考：

- Linear：任务、状态流和工程效率工具。
- Notion：文档产物、轻量编辑和知识沉淀。
- Figma / Miro：白板、画布和多端协作。
- Vercel / Cursor：开发者工具式的清晰仪表盘。

不要直接复制品牌视觉。只抽取适合本项目的设计原则和组件密度。

## 6. 本项目的固定验证命令

每次提交前至少运行：

```powershell
npm run typecheck
npm run build
npm run test:e2e
```

涉及飞书 CLI 时额外运行：

```powershell
lark-cli doctor
lark-cli auth status
```

涉及真实飞书写操作时，必须记录：

- 操作对象：测试群 / 测试文档 / 测试文件夹。
- 是否发出真实消息或创建真实文档。
- 产物链接是否可打开。

## 7. 第二周期开发原则

第二周期只追求一个真实小闭环：

```text
测试群 IM 消息
  -> 豆包 2.0 Pro 规划和生成
  -> 创建真实飞书 Docs
  -> 回发文档链接到测试群
  -> Web 仪表盘同步展示
```

Slides、自由画布、原生移动端、离线同步都先不抢跑。它们进入第三阶段或第二周期后半段，前提是 Docs 小闭环已经稳定。

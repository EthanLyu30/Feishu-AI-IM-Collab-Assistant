# 第二周期任务规划

周期建议：2026.04.26 - 2026.05.02

## 0. 第二周期目标

第二周期的目标不是继续堆功能，而是把第一周期的 Mock Demo 推进成一个可演示的真实飞书小闭环。

核心目标：

```text
测试群 IM 消息
  -> Agent 读取上下文
  -> 豆包 2.0 Pro 生成计划和需求文档
  -> 创建真实飞书 Docs
  -> 将文档链接回发到测试群
  -> Web 仪表盘展示任务状态和产物链接
```

本周期暂不追求完整真实 PPT / 自由画布生成，优先把 IM + Doc + Agent + 多端仪表盘这条链路做稳。

## 1. 开发流程对齐检查

参考 `andrej-karpathy-skills` 的四条原则，当前项目流程对齐情况如下。

| 原则 | 当前情况 | 问题 | 第二周期调整 |
| --- | --- | --- | --- |
| 编码前思考 | 已做赛题拆解、架构规划、Demo 脚本 | 部分真实飞书接入仍有隐含假设，例如 CLI 输出结构和 chat_id 来源 | 每个真实 API 接入前先写成功标准和验证命令 |
| 简洁优先 | 先做 Mock 主链路，再接真实能力，方向正确 | 第一周期 scaffold 较大，后续容易继续扩散 | 第二周期只做真实 Docs 小闭环，不抢跑 Slides / 画布 |
| 精准修改 | 大部分改动围绕比赛主链路 | 曾出现临时硬编码 Mock 上下文的改动，需要避免类似绕过 Adapter 的做法 | 后续每次提交前检查 diff，确认每行都对应当前任务 |
| 目标驱动执行 | 已有 typecheck、build、E2E | 真实飞书操作还缺手动验证记录 | 每个真实飞书任务必须记录测试资源和产物链接 |

结论：

当前开发方向基本符合，但执行粒度还需要更严格。第二周期开始，每个任务都要按：

```text
目标 -> 假设 -> 最小实现 -> 验证 -> 记录
```

这个格式推进。

## 2. 任务一：确认测试资源

目标：

准备一个不会影响正式协作的飞书测试群和测试文档空间。

需要确认：

- 测试群名称。
- 测试群 `chat_id`。
- 是否允许 Agent 在群里发送测试消息。
- 是否需要指定 Docs 创建目录 `folder_token`。

成功标准：

- `.env` 中配置 `LARK_DEFAULT_CHAT_ID`。
- `lark-cli im +chat-messages-list` 可以读取测试群最近消息。
- `lark-cli im +messages-send` 可以向测试群发送一条简单测试消息。

验证命令：

```powershell
lark-cli doctor
lark-cli im +chat-search --query "测试群关键词" --format pretty
lark-cli im +chat-messages-list --chat-id "oc_xxx" --page-size 5 --format pretty
lark-cli auth login --scope "im:message.send_as_user"
lark-cli im +messages-send --as user --chat-id "oc_xxx" --text "Agent-Pilot 回发验证"
```

注意：

真实群聊写入前必须确认是测试群。当前已验证测试群可读、真实 Docs 可创建；群回发还需要补齐 `im:message.send_as_user` 授权。

## 3. 任务二：豆包 2.0 Pro 接入稳定化

目标：

让 Planner、文档生成、PPT 大纲生成、追加修改默认走豆包 2.0 Pro，并保留 Mock 回退能力。

最小实现：

- 保持 `AGENT_LLM_MODE=mock` 可稳定通过 E2E。
- 在本地 `.env` 中使用 `AGENT_LLM_MODE=doubao` 手动验证真实生成。
- 为 Planner 输出增加 Schema 校验。
- JSON 解析失败时重试一次。
- 失败后给出清晰错误，不静默吞掉。

成功标准：

- Mock 模式：`npm run test:e2e` 通过。
- Doubao 模式：手动创建任务能生成非模板化计划和文档。
- Planner 输出不符合结构时有可读错误。

建议文件：

```text
apps/api/src/agent/schemas.ts
apps/api/src/agent/Verifier.ts
apps/api/src/llm/json.ts
```

## 4. 任务三：真实飞书 Docs 创建闭环

目标：

把 Agent 生成的需求文档创建为真实飞书文档，并把真实链接写入 Artifact。

当前基础：

`LarkCliAdapter.createDoc` 已开始使用临时 Markdown 文件调用：

```powershell
lark-cli docs +create --as user --title ... --markdown @file
```

需要完善：

- 稳定解析 CLI 输出。
- 如果 CLI 输出没有链接，保留原始输出供排查。
- 创建失败时返回明确错误。
- 临时文件写入 `.tmp/`，不进入 Git。

成功标准：

- `OFFICE_ADAPTER=lark-cli` 时，能创建真实飞书 Docs。
- 前端产物卡片中显示真实飞书文档链接。
- Mock 模式 E2E 仍通过。

验证：

```powershell
npm run typecheck
npm run build
npm run test:e2e
```

手动验证：

```powershell
$env:OFFICE_ADAPTER="lark-cli"
$env:AGENT_LLM_MODE="doubao"
npm run dev
```

## 5. 任务四：测试群回发交付摘要

目标：

Agent 生成文档后，把交付摘要和 Docs 链接回发到飞书测试群。

最小实现：

- 仅在配置了 `LARK_DEFAULT_CHAT_ID` 时回发。
- 回发内容只包含标题、文档链接和下一步建议。
- 失败不影响主任务完成，但要在事件日志中记录。

成功标准：

- 测试群收到 Agent 回发消息。
- Web 仪表盘仍显示任务完成。
- 回发失败时有清晰错误事件。

暂不做：

- 群内自动监听事件。
- @机器人触发。
- 多群路由。

## 6. 任务五：Web 仪表盘最小增强

目标：

让 Demo 更容易展示真实飞书小闭环。

最小实现：

- 增加当前运行模式显示：`mock` / `doubao` / `lark-cli`。
- 产物卡片支持打开真实链接。
- 增加任务事件中的错误提示。
- 增加“测试群 chat_id 已配置 / 未配置”状态。

成功标准：

- 演示时能看懂当前是 Mock 还是真实飞书链路。
- 文档链接可以直接打开。
- 手机视图和桌面视图不重叠、不溢出。

验证：

```powershell
npm run test:e2e
```

再手动打开：

```text
http://localhost:5173
```

## 7. 任务六：阶段材料沉淀

目标：

第二周期末能直接写阶段小结和 Demo 汇报。

每天记录：

- 当天完成的真实链路。
- 使用豆包的场景。
- 飞书 CLI 调用次数或关键命令。
- 遇到的错误和解决方式。
- 生成的真实文档链接。

第二周期小结建议指标：

| 指标 | 记录方式 |
| --- | --- |
| 真实飞书文档数 | Docs 创建成功次数 |
| 真实测试群回发次数 | IM 回发成功次数 |
| 豆包调用场景 | Planner / Doc / PPT / 修改 |
| 自动化验证次数 | typecheck / build / E2E |
| 端到端耗时 | 从 IM 指令到文档链接回发 |

## 8. 不做清单

第二周期前半段暂不做：

- 原生 iOS / Android。
- Electron / Tauri 打包。
- 完整 Slides XML 设计器。
- 离线 CRDT 合并。
- 富媒体图片 / 表格自动插入。
- 多群、多租户、多用户权限系统。

原因：

这些能力都重要，但会稀释当前最关键的真实飞书小闭环。先把 IM + Agent + Docs 跑稳，再扩展 PPT 和多端。

## 9. 第二周期验收标准

本周期完成时，至少应该能演示：

```text
1. 测试群里存在一段需求讨论。
2. Web 仪表盘触发 Agent 任务。
3. Agent 读取测试群上下文。
4. 豆包 2.0 Pro 生成结构化需求文档。
5. 系统创建真实飞书 Docs。
6. Agent 把文档链接回发到测试群。
7. Web 仪表盘展示任务进度、文档链接和交付摘要。
```

如果这条链路稳定，再进入 Slides / 自由画布。

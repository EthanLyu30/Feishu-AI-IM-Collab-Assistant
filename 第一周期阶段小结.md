## 三、过程复盘与沉淀（必填）

### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

本周期主要完成了四类工作。

第一类是赛题理解和方案收敛。  
比赛题目要求覆盖 IM、文档、PPT / 自由画布、多端协同、自然语言交互和 Agent 编排，范围比较大。我先把项目定位确定为“AI Agent 是 Pilot，GUI 是 Co-pilot”，避免把作品做成普通聊天机器人或单纯 GUI 工具。随后把完整能力拆成 A-F 六个模块化场景：IM 入口、任务规划、文档生成、PPT / 画布生成、多端协同、总结交付。

第二类是工程底座搭建。  
我搭建了 TypeScript monorepo，拆分为：

- `apps/api`：Agent 后端、Planner、内容生成、Office Adapter、WebSocket 同步。
- `apps/web`：React + Vite 多端协同仪表盘。
- `packages/shared`：任务、步骤、事件、产物等共享类型。

当前本地 Demo 已经能模拟飞书 IM 讨论，并自动生成需求文档、PPT 内容、3 分钟讲稿和交付摘要。Web 页面左侧模拟移动端 IM 入口，右侧模拟桌面端 Agent 仪表盘，体现多端协同状态。

第三类是豆包 2.0 Pro 和飞书接入准备。  
豆包侧已经预留 Ark Provider，后续可通过 `.env` 切换：

```env
AGENT_LLM_MODE=doubao
```

飞书侧完成了 `larksuite/cli` 的安装、App 配置、用户授权和 `LARK_CLI_NO_PROXY=1` 设置。现在 CLI 可以用用户身份访问飞书 API，并且已确认具备 IM、Docs、Slides、Whiteboard、Drive 等权限。在工程上，`LarkCliAdapter` 已经开始承接真实飞书能力：读取群消息需要配置 `LARK_DEFAULT_CHAT_ID`，创建 Docs 会通过临时 Markdown 文件调用 `lark-cli docs +create`。这部分还属于第一版接入骨架，后续需要继续完善返回值解析、错误处理、Slides XML 生成和 IM 回发链路。

第四类是验证和沉淀。  
我增加了 Playwright E2E 测试，用浏览器真实点击验证“IM 指令 -> Agent 规划 -> 文档 -> PPT -> 讲稿 -> 追加修改”的主链路。除此之外，还产出了项目技术方案、Demo 演示脚本、后续开发指导书和阶段小结素材，方便后续继续迭代和准备比赛汇报。

AI 辅助主要用于：

### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

本周期最明显的卡点有三个。

第一，飞书 CLI 授权链路比预期复杂。  
一开始以为拿到 App ID 和 App Secret 就能直接操作飞书，但实际流程还包括应用配置、权限范围、企业审批、用户授权和 token 校验。中间先后遇到了 `client secret invalid`、`app pending approval`、用户 token 不存在等问题。最后通过逐步使用 `lark-cli auth scopes`、`lark-cli auth login`、`lark-cli doctor` 排查，才确认 App Secret、应用审批和用户授权都完成了。这次也让我明确了：App ID / App Secret 只代表应用身份，不等于应用已经被企业批准，也不等于用户已经授权。

第二，赛题范围很大，容易失焦。  
题目涉及 IM、Doc、PPT / 画布、多端同步、Agent 编排和自然语言交互，如果一开始就试图把所有能力都做完整，很容易变成多个半成品。破局方式是先确定一条最短闭环：IM 指令 -> Agent 规划 -> 文档 -> PPT -> 交付；再把其他能力作为可组合模块逐步接入。

第三，真实飞书写操作需要谨慎推进。  
目前 CLI 已经具备操作飞书的能力，但真实写文档、发群消息、创建 Slides 都会影响真实飞书空间。为了降低风险，当前阶段没有直接把所有真实写操作接进演示主链路，而是先在 `LarkCliAdapter` 中做渐进式接入：先支持读取固定测试群消息和创建飞书 Docs，再逐步扩展更新文档、生成 Slides XML 和回发群消息。这样能避免为了追求“全真链路”而破坏已经稳定的 Mock Demo。

### 3. 有没有什么这次写得特别顺，或者下次还能直接复用的东西？

这次最值得复用的是 Adapter-first 架构和主链路验证方式。

通过 `OfficeToolAdapter` 抽象飞书办公能力，项目可以先用 Mock 数据演示完整 Agent 编排，再逐步切换到真实 `lark-cli` 或飞书开放平台 API。这样即使后续 API 能力、权限、命令格式发生变化，也不会推翻 Agent 主流程。

可复用内容包括：

- Agent 场景拆解模板：入口、规划、执行、同步、交付。
- Agent 工程模块：Planner、Content Composer、Office Adapter、Task Store、Realtime Event。
- Planner Prompt 模板：强制输出 JSON，包含 goal、steps、requiredConfirmations、risks。
- Demo 主线模板：自然语言触发 -> 任务规划 -> 产物生成 -> 用户修改 -> 交付归档。
- Playwright E2E 验证动作：点击 IM 发送按钮，等待文档、PPT、讲稿和追加修改内容出现。
- 飞书 CLI 排障流程：先 `config init`，再 `auth scopes`，再 `auth login`，最后用 `doctor` 验证。
- 安全处理方式：真实 API Key、App Secret 只放本地环境变量或 CLI 配置，不写入 GitHub。
- 真实飞书接入方式：长 Markdown 内容通过临时文件传给 `lark-cli`，避免命令行转义和长度问题。

## 四、本周随手记（选填）

- 这个赛题的重点不是“生成一份文档”，而是展示 Agent 如何理解任务、拆解任务、调用工具并交付结果。
- GUI 应该服务于 Agent，而不是抢走主流程。当前 Web 仪表盘的价值是让评委看见 Agent 在做什么、做到哪一步、产出了什么。
- 豆包 2.0 Pro 接入后，要优先保证结构化输出稳定。Planner 不稳定，后面的文档和 PPT 都会不稳定。
- 飞书 Docs 是最适合优先真实接入的办公组件；Slides 更复杂，因为 `lark-cli slides +create` 需要 XML slide array。
- IM 自动回复最好先在测试群验证，避免真实群聊被频繁测试消息打扰。
- `LARK_CLI_NO_PROXY=1` 可以只让飞书 CLI 不走代理，同时不影响 Codex 对话继续走代理。
- 第一周期不追求功能全，而是先把“方向正确、链路可跑、接入可继续”的地基搭牢。
- 后续最优先的三件事是：切换豆包 2.0 Pro、配置测试群 `chat_id` 跑通真实 IM 读取、完善 Docs 创建后的链接解析与群聊回发。

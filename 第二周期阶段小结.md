## 一、核心产出（必填）

本周期我交付的最满意成果是：

> 将 Agent-Pilot 从第一周期的 Mock 原型推进成了一个真实飞书小闭环：Agent 能读取测试群讨论，调用豆包 2.0 Pro 进行任务规划和内容生成，自动创建真实飞书 Docs 和 Slides，并把交付结果回发到飞书群，同时 Web 仪表盘可以同步展示任务状态、产物链接和错误信息。

这个成果对我来说最有意义的地方在于，项目开始从“能演示的想法”变成“能驱动真实办公套件的系统”。第一周期更像是在搭舞台：确定赛题方向、搭建工程底座、跑通 Mock 链路；第二周期则开始真正接触飞书生态里的真实能力，包括群消息、文档、演示稿、回发消息和用户确认。这一步之后，作品不再只是一个漂亮的前端页面，而是有了可以被评委追问和验证的工程闭环。

从比赛要求看，这个阶段已经初步体现了“AI Agent 是主驾驶，GUI 是辅助仪表盘”的设计理念。用户可以通过自然语言提出需求，Agent 先理解并拆解任务，再调用飞书相关能力完成文档、演示稿和交付摘要生成。Web 仪表盘不是最终产物本身，而是开发期和演示期用于观察 Agent 状态、产物和错误的控制台；最终形态仍应包装成飞书应用内页面，让群聊负责触发，让页面负责展示进度和确认节点。

本周期形成的核心闭环如下：

```text
飞书测试群讨论 / Web 指令
  -> Agent 读取上下文
  -> 豆包 2.0 Pro 规划任务
  -> 生成需求文档
  -> 创建真实飞书 Docs
  -> 生成汇报 Slides 内容
  -> 创建真实飞书 Slides
  -> 生成 3 分钟讲稿和交付摘要
  -> 回发链接到飞书测试群
  -> Web 仪表盘实时同步任务状态
```

另外，本周期还参考 Hermes-Agent 的开发方式，把飞书 IM 入口从“收到消息就执行的一次性 webhook”推进为更接近真实 Agent 的会话流程：

```text
飞书 IM 事件
  -> Lark IM trigger adapter
  -> 绑定 chatId 对应任务会话
  -> Planner 生成执行计划
  -> waiting_user 等待群内确认
  -> 用户回复“确认 / 继续”
  -> Office tools 执行 Docs / Slides / 回发
```

这个设计让我对“Agent 驱动”有了更具体的理解：Agent 不是简单地把一个按钮换成一句自然语言，而是要能理解上下文、保存会话状态、在关键节点请求确认，并把工具调用结果回到原来的协作场景中。这个认识会直接影响后续的产品设计和技术实现。

## 三、过程复盘与沉淀（必填）

### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

本周期主要完成了五类工作。

第一类是真实飞书能力接入。  
第一周期已经完成飞书 CLI 授权和基础适配器雏形，本周期重点把 `LarkCliAdapter` 打磨成可支撑真实 Demo 的工具边界。现在它已经支持读取测试群消息、创建真实飞书 Docs、更新飞书 Docs、创建真实飞书 Slides，并且可以向测试群回发交付摘要。长 Markdown 内容通过 `.tmp/` 临时文件传给 `lark-cli`，避免命令行长度、空格和转义问题；飞书返回内容则通过统一的 JSON / 文本 URL 提取逻辑写回 Artifact。

第二类是豆包 2.0 Pro 生成链路稳定化。  
为了避免 LLM 输出格式不稳定影响后续工具调用，本周期给 Planner 增加了 Zod Schema 校验，并限制步骤工具只能来自已支持的工具集合。如果豆包第一次输出不符合结构，系统会自动重试一次；如果重试仍失败，就返回可读错误，而不是让后续执行吃到脏数据。这部分让我意识到，LLM 接入不是“能调通接口”就算完成，真正有工程价值的是把不稳定的自然语言输出约束成可校验、可重试、可降级的系统行为。

第三类是真实 Slides 生成。  
原本第二周期不打算重点做完整 PPT，但在跑通 Docs 后，我继续补上了 Slides 创建能力。过程中发现飞书 Slides 不能直接接受普通 Markdown 或简单 `<slide><text>...</text></slide>` 结构，否则会生成空白 PPT。后来通过读取和验证飞书 Slides XML，确认需要使用 SML 2.0 中的文本框结构，将内容写成 `<shape type="text"><content><p>...</p></content></shape>`。基于这个规则，我新增了 `SlidesXmlBuilder`，把 Markdown 大纲转换为可渲染的封面页和内容页。

第四类是 IM 触发和 Agent 会话化。  
本周期新增了 `/api/triggers/lark-im` 和 `/api/lark/events` 两个入口，支持飞书 challenge、扁平测试 payload、飞书事件形状 payload、关键词触发和 `messageId` 幂等去重。后续又参考 Hermes-Agent 的 Gateway / Session / Tool Loop 思路，补上了群内确认机制：IM 触发后任务先进入 `waiting_user`，Agent 回发规划和确认请求，用户在同一群聊回复“确认 / 继续”后，系统再继续生成 Docs、Slides 和交付摘要。

第五类是验证、记录和演示材料沉淀。  
本周期持续使用 `npm run typecheck`、`npm run build`、`npm run test:e2e` 作为提交前验证，并用真实飞书测试群手动验证 `lark-cli doctor`、群消息读取、Docs 创建、Slides 创建和群回发。文档方面，更新了第二周期执行记录、赛题完成度检查、飞书 IM 触发接入说明、后续开发指导书和开发范式，为后续继续开发和阶段汇报保留依据。

AI 辅助主要用于：

- 帮助对照赛题要求，判断第二周期优先级。
- 协助设计 `OfficeToolAdapter`、`LarkCliAdapter`、`AgentOrchestrator` 和 IM trigger 的模块边界。
- 生成和修改 TypeScript 后端、React 仪表盘、E2E 测试和 Markdown 文档。
- 辅助排查飞书 CLI 参数、消息结构、Slides XML 结构、群回发格式和 `.env` 覆盖问题。
- 借鉴 Hermes-Agent 的会话网关思路，把 IM 入口从简单触发改成可确认、可继续的 Agent 会话。

自己主要负责：

- 提供真实飞书应用、测试群和授权信息。
- 判断真实写操作是否允许在测试群中执行。
- 决定第二周期仍以真实飞书小闭环为核心，而不是过早扩展原生客户端和离线能力。
- 关注比赛验收中 IM、Doc、Slides 和多端仪表盘之间的演示叙事。
- 在多次调试中判断哪些问题必须当场修复，哪些应放入下一阶段，例如公网部署、内嵌页面和持久化。

这一周期我最明显的感受是，AI 可以非常快地帮我铺开代码和方案，但真正决定项目能不能跑起来的，是我能不能把问题拆到足够小，并且一层层验证。很多时候，错误不是出在“AI 生成得不够多”，而是出在我一开始对真实平台的输出结构、权限边界或数据格式假设得太顺了。第二周期让我更愿意先做小验证，再把能力接进主链路。

### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

本周期遇到的主要卡点有五个。

第一，飞书群消息一开始读出来是 0 条。  
原因不是测试群没有消息，而是 `lark-cli im +chat-messages-list --format json` 的真实输出结构与最初假设不同。代码一开始主要按 `items` 解析，但真实结果在 `data.messages` 中。后来通过检查 CLI 输出结构，修正 `LarkCliAdapter.readMessages` 的解析逻辑，同时过滤已删除消息和空内容，最终读群从 0 条修复到 10 条有效讨论消息。这个问题让我印象比较深，因为它提醒我：真实 API 接入时，必须相信实际返回，不要相信自己脑中“应该长这样”的结构。

第二，文档追加修改一开始提示 `LarkCliAdapter.updateDoc is not wired yet`。  
这说明 GUI 上已经有“追加修改”的入口，但真实飞书文档更新工具还没有接上。后来补齐了 `lark-cli docs +update --mode overwrite` 调用，将自然语言追加修改生成的新 Markdown 覆盖到原文档中，并更新 Artifact 版本号。这个问题也提醒我：Web 上出现的按钮必须对应真实工具能力，否则会让 Demo 看起来像半成品。

第三，飞书 Slides 生成后打开是空白。  
这个问题卡得比较典型，因为“创建成功”和“内容可见”不是一回事。最开始只确认了 Slides 链接能打开，但打开后页面没有内容。后来通过反查和读取 Slides XML，发现飞书 Slides 需要特定的 SML 结构，不能把文本直接塞进普通标签。最后新增 `SlidesXmlBuilder`，用文本框、段落和背景样式构造页面，才让生成的 Slides 真正显示标题和要点。解决这个问题之后，我对“验收标准”也更谨慎了：不能只看接口是否返回成功，还要看用户最终看到的结果是否真的可用。

第四，群回发 Markdown 链接不稳定。  
飞书 IM 对富文本链接格式和 URL 校验比较严格，直接回发 Markdown 链接时容易触发格式问题。为了保证第二周期闭环稳定，当前把群回发改成纯文本摘要，把标题和 URL 直接写出来。这个方案不够漂亮，但稳定性高，适合当前阶段演示。这也让我接受了一个工程取舍：比赛阶段先保证主链路可靠，再逐步追求消息格式和视觉表现。

第五，测试环境曾被本地真实 `.env` 干扰。  
E2E 应该走 Mock 模式，但本地 `.env.local` 或真实配置可能覆盖 `AGENT_LLM_MODE=mock`、`OFFICE_ADAPTER=mock`，导致自动化测试误连真实飞书或豆包。后来调整环境变量加载策略，让显式传入的 E2E mock 配置优先级更高，保证自动化测试稳定可重复，同时真实飞书配置仍只用于手动验证。这个坑让我进一步意识到，自动化测试不只是检查功能，也是在保护真实环境不被误操作。

### 3. 有没有什么这次写得特别顺，或者下次还能直接复用的东西？

这次最值得复用的是“Adapter-first + 事件驱动 + 会话确认”的实现套路。

`OfficeToolAdapter` 让 Mock 和真实飞书能力使用同一套 Agent 编排逻辑，后续即使从 `lark-cli` 切到飞书开放平台 API，也可以优先替换 Adapter，而不必重写 Planner、Content Composer 和 Web 仪表盘。这个结构让我在调试时比较有安全感：Mock 链路可以持续兜底，真实飞书能力则可以逐个工具替换和验证。

可复用内容包括：

- 真实飞书接入模式：先 CLI dry-run 或单工具验证，再接入 Agent 主链路。
- 长文档写入方式：Markdown 写入 `.tmp/`，通过 `--markdown @file` 传给 `lark-cli`。
- CLI 输出解析方式：先尝试 JSON 解析，再递归查找 URL，最后回退文本扫描。
- Planner 稳定化方式：LLM 输出必须经过 Schema 校验，失败后重试一次。
- Slides 生成经验：飞书 Slides 需要 SML 2.0 结构，文本应放进 shape/content/p。
- IM 触发处理方式：抽取 payload、识别关键词、按 `messageId` 去重、按 `chatId` 找任务会话。
- 群内确认机制：先 `waiting_user`，再由“确认 / 继续”消息恢复执行，避免 Agent 擅自生成正式产物。
- 验证命令组合：`npm run typecheck`、`npm run build`、`npm run test:e2e` 加真实 `lark-cli doctor` 和手动飞书验证。
- 安全处理动作：真实 API Key、App Secret、测试群 ID、文档 token 只放本地，不写进 Git。

这一套方法后续还能直接用于接飞书事件订阅、富文本消息、语音消息、图片 / 表格处理和飞书内嵌页面。只要保持“先验证单工具，再接入 Agent 编排，再补 UI 展示”的节奏，项目就不容易因为功能扩展而失控。

## 四、本周随手记（选填）

- 这一周期最强的感受是：真实平台接入会把很多“看起来已经完成”的功能重新拉回地面。Mock 能证明思路，真实飞书才能证明工程落地。
- App ID 和 App Secret 只是应用身份，不代表已经具备所有操作能力；真正能不能操作还取决于权限、审批、用户授权和 token 状态。
- 飞书 CLI 的真实输出结构要以实际命令结果为准，不能只按想象中的 API 字段写解析。
- Slides 最容易出现“链接创建成功但内容不可见”的假成功，后续每次改 Slides 都要打开或读取 XML 验证。
- Web 仪表盘目前是开发期控制台，不是最终产物。最终更合理的形态是嵌入飞书应用内，让群聊负责触发，页面负责展示进度、产物和确认节点。
- Agent 驱动不是“收到命令就自动做完所有事”，关键节点应该请求确认，尤其是会创建真实文档、真实演示稿或向群里发消息时。
- 第二周期已经证明 TypeScript 作为主栈是合适的：后端、前端、共享类型、WebSocket、LLM 接口和飞书 CLI 都能统一在一个 monorepo 里推进。

# 第三周期执行记录

更新时间：2026.05.01

## 1. 本轮目标

按照第三周期规划，优先推进 Hermes-style IM Agent 会话基础设施：

```text
IM 事件网关
  -> 安全过滤与幂等
  -> chatId 会话状态
  -> 群内确认 / 进度 / 取消
  -> 工具执行和原群交付
```

本轮不再把目标停留在“后端会话层可用”，而是继续推进到飞书应用可演示入口、稳定 API 路线和 Agent 运行质量。当前策略是：飞书事件仍优先走官方长连接，仪表盘和 API 访问通过 Cloudflare Pages / Named Tunnel 固定入口。

## 2. Agent 团队分工

本轮采用 6 个子 agent 并行分析：

| 角色 | 输出重点 |
| --- | --- |
| 产品经理 | 收敛 P0 演示链路：IM 触发、会话状态、工具执行、原群交付 |
| 需求分析 | 对照赛题 Must-have / Good-to-have，确认第三周期优先缺口 |
| 后端安全 | 检查 verify token、签名、白名单群、自消息过滤和幂等风险 |
| 持久化 / 会话状态 | 设计任务、事件、handled message IDs 的最小持久化方向 |
| 前端 / 内嵌页面 | 检查 Web 仪表盘如何改造成飞书内嵌演示页 |
| 测试 | 梳理 Hermes-style IM 网关与会话状态的测试缺口 |

主线程负责集成和落地本轮最小代码变更。

## 3. 已完成

### 3.1 飞书事件入口安全守卫

新增：

```text
apps/api/src/security/LarkEventGuard.ts
```

完成：

- 支持可选 `LARK_EVENT_VERIFY_TOKEN` 校验。
- 支持可选 `LARK_EVENT_ENCRYPT_KEY`，按飞书 `X-Lark-Signature` 规则校验 raw body。
- Express JSON parser 已保留 `rawBody`，避免 parse / stringify 后签名不一致。
- 支持 `LARK_ALLOWED_CHAT_IDS` 白名单群过滤。
- 支持 `LARK_BOT_OPEN_ID` / `LARK_BOT_USER_ID` 和 `senderType=app/bot` 机器人自消息过滤。
- 对被忽略消息返回明确 reason，便于演示和排查。

当前签名校验代码已就绪，后续公网事件订阅时需要在真实飞书后台配置中验证。

### 3.2 messageId 持久化去重

新增：

```text
apps/api/src/state/HandledMessageStore.ts
```

完成：

- 将原来的内存 `Map<string, string>` 替换为 `HandledMessageStore`。
- 默认写入 `.data/lark-state.json`，`.data/` 已加入 `.gitignore`。
- E2E 模式通过 `LARK_STATE_PATH=:memory:` 使用内存态，避免测试之间互相污染。
- 同一个 `messageId` 重复进入时仍返回 `duplicate message ignored`。

当前只持久化 handled message IDs，还没有持久化 Task、Event 和 Artifact。

### 3.3 群内会话命令扩展

完成：

- 新增 `isCancelText`，支持“取消 / 停止 / 终止 / cancel”等指令。
- 新增 `isProgressText`，支持“进度 / 状态 / status / progress”等指令。
- `AgentOrchestrator` 新增 `cancelTask` 和 `reportTaskProgress`。
- `TaskStatus` 新增 `cancelled`。
- 取消任务会写入 `task.cancelled` 事件。
- 查询进度会返回当前任务状态、步骤完成数和已生成产物。

### 3.4 同群活跃会话互斥

完成：

- 同一个 `chatId` 已存在 `created / planning / waiting_user / running` 任务时，新的触发消息不再创建新任务。
- 返回 reason：`chat session already active`。

这个改动避免了 Hermes-style per-chat session 中最容易出现的“一个群挂多个 waiting 任务”问题。

### 3.6 飞书机器人菜单重排方案

已明确当前截图里“打开仪表盘”没有出现的原因：机器人一级菜单已被“开始整理 / 查看进度 / 取消任务”占满。新的配置方案已写入：

```text
飞书开放平台后台填写清单.md
飞书应用初版交付与部署指南.md
飞书开放平台长连接接入指南.md
README.md
```

推荐菜单：

```text
任务 -> 开始整理 / 查看进度 / 取消任务
仪表盘 -> 跳转 https://feishu-ai-im-collab-assistant.pages.dev
帮助 -> 发送使用说明
```

如果后台只配 3 个一级菜单，则使用：

```text
开始整理 / 打开仪表盘 / 取消任务
```

`查看进度` 由用户直接在群内发送 `进度` 完成，不再占一级菜单位。修改菜单后必须在飞书开放平台创建版本并发布，客户端侧可能有数分钟缓存。

### 3.8 Agent 运行质量增强

新增：

```text
apps/api/src/agent/ArtifactVerifier.ts
```

完成：

- Docs 生成后检查内容长度、标题层级和角色 / 权限覆盖。
- Slides 生成后检查页数、内容密度和讲稿 / 汇报口径。

- 校验失败不会阻断演示主链路，而是通过 `integration.warning` 给出可解释建议。

### 3.9 任务持久化与 CLI 稳定性

新增能力：

- `TaskStore` 支持把 Task、Event、Artifact 持久化到 `.data/tasks.json`。
- API 重启后仍可恢复最近任务、产物和事件。
- `LarkCliAdapter` 增加命令超时，默认 `45000ms`。
- 飞书消息读取增加只读重试，默认 1 次；创建 Docs / Slides / 群回发不做自动重试，避免重复创建或重复发送。
- `.env.example` 补充 `TASK_STATE_PATH`、`LARK_CLI_TIMEOUT_MS`、`LARK_CLI_READ_RETRIES`。

下一步建议按顺序推进：

1. 完成 `npm run build` 和 `npm run test:e2e`，确认新增持久化与 Verifier 不破坏主链路。

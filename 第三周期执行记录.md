# 第三周期执行记录

更新时间：2026.04.28

## 1. 本轮目标

按照第三周期规划，优先推进 Hermes-style IM Agent 会话基础设施：

```text
IM 事件网关
  -> 安全过滤与幂等
  -> chatId 会话状态
  -> 群内确认 / 进度 / 取消
  -> 工具执行和原群交付
```

本轮不追求完整公网部署和飞书内嵌页面，先把后端会话层补稳，避免真实飞书事件接入后出现重复触发、自触发循环或同群多个悬挂任务。

## 2. Agent 团队分工

本轮采用 6 个子 agent 并行分析：

| 角色 | 输出重点 |
| --- | --- |
| 产品经理 | 收敛 P0 演示链路：IM 触发、会话状态、工具执行、原群交付 |
| 需求分析 | 对照赛题 Must-have / Good-to-have，确认第三周期优先缺口 |
| 后端安全 | 检查 verify token、签名、白名单群、自消息过滤和幂等风险 |
| 持久化 / 会话状态 | 设计任务、事件、handled message IDs 的最小持久化方向 |
| 前端 / 内嵌页面 | 检查 Web 仪表盘如何改造成飞书内嵌演示页 |
| 测试 | 梳理 Hermes-style IM 网关与会话状态的测试缺口 |

主线程负责集成和落地本轮最小代码变更。

## 3. 已完成

### 3.1 飞书事件入口安全守卫

新增：

```text
apps/api/src/security/LarkEventGuard.ts
```

完成：

- 支持可选 `LARK_EVENT_VERIFY_TOKEN` 校验。
- 支持 `LARK_ALLOWED_CHAT_IDS` 白名单群过滤。
- 支持 `LARK_BOT_OPEN_ID` / `LARK_BOT_USER_ID` 和 `senderType=app/bot` 机器人自消息过滤。
- 对被忽略消息返回明确 reason，便于演示和排查。

当前仍未完成完整 `X-Lark-Signature` raw body 签名校验，后续公网事件订阅前需要继续补。

### 3.2 messageId 持久化去重

新增：

```text
apps/api/src/state/HandledMessageStore.ts
```

完成：

- 将原来的内存 `Map<string, string>` 替换为 `HandledMessageStore`。
- 默认写入 `.data/lark-state.json`，`.data/` 已加入 `.gitignore`。
- E2E 模式通过 `LARK_STATE_PATH=:memory:` 使用内存态，避免测试之间互相污染。
- 同一个 `messageId` 重复进入时仍返回 `duplicate message ignored`。

当前只持久化 handled message IDs，还没有持久化 Task、Event 和 Artifact。

### 3.3 群内会话命令扩展

完成：

- 新增 `isCancelText`，支持“取消 / 停止 / 终止 / cancel”等指令。
- 新增 `isProgressText`，支持“进度 / 状态 / status / progress”等指令。
- `AgentOrchestrator` 新增 `cancelTask` 和 `reportTaskProgress`。
- `TaskStatus` 新增 `cancelled`。
- 取消任务会写入 `task.cancelled` 事件。
- 查询进度会返回当前任务状态、步骤完成数和已生成产物。

### 3.4 同群活跃会话互斥

完成：

- 同一个 `chatId` 已存在 `created / planning / waiting_user / running` 任务时，新的触发消息不再创建新任务。
- 返回 reason：`chat session already active`。

这个改动避免了 Hermes-style per-chat session 中最容易出现的“一个群挂多个 waiting 任务”问题。

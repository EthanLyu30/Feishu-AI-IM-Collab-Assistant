# 第三周期任务规划

周期建议：2026.04.29 - 2026.05.05

## 0. 第三周期目标

第三周期的目标不是继续扩大 Mock Demo，而是把第二周期已经跑通的“本地真实飞书小闭环”推进成“飞书内可演示、可恢复、可解释”的产品雏形。

第二周期已经完成：

```text
测试群 / Web 指令
  -> Agent 读取上下文
  -> 豆包 2.0 Pro 规划和生成
  -> 创建真实飞书 Docs
  -> 创建真实飞书 Slides
  -> 回发交付摘要
  -> Web 仪表盘同步展示
```

第三周期的核心目标是：

```text
飞书群内 @Agent / 指令触发
  -> 飞书长连接事件进入本地 Agent 运行时
  -> 去重、白名单和机器人自消息过滤
  -> Agent 规划并在群内请求确认
  -> 用户回复确认 / 修改 / 取消
  -> Agent 生成真实 Docs + Slides + 讲稿
  -> 飞书内嵌页面实时展示进度和产物
  -> 群聊收到最终交付链接和下一步建议
```

本周期完成后，项目应具备一条更贴近比赛验收的演示路径：移动端或桌面端飞书群聊触发，飞书开放平台通过长连接把真实消息事件推给本地 Agent，Agent 自动执行，飞书应用内页面展示进度和产物，最终回到群聊交付。

## 1. 开发范式要求

第三周期继续遵循 `开发范式.md` 中的原则：

```text
目标 -> 假设 -> 最小实现 -> 验证 -> 记录
```

每个真实飞书相关任务都必须先写清：

- 操作对象是哪个测试资源。
- 是否会发送真实群消息。
- 是否会创建或更新真实 Docs / Slides。
- 成功标准是什么。
- 失败后如何回滚或降级。

本周期新增原则：

- 不再把 Web 仪表盘当成最终形态，而是把它包装成飞书内嵌页面。
- 不再只验证接口返回成功，必须验证用户最终可见结果，例如 Docs 有内容、Slides 非空白、群消息可读。
- 不在代码或文档中提交真实 App Secret、Ark API Key、测试群 ID、文档 token。
- 所有飞书事件入口必须考虑幂等、白名单和机器人自消息过滤；如果后续切到 webhook，再启用 raw body 签名校验。

## 2. 任务一：飞书后台配置与长连接事件接入

目标：

让飞书后台中的机器人、菜单、事件订阅和本地长连接运行时对齐，使项目不再依赖手动调用本地 API 模拟 IM 触发。

最小实现：

- 按 `飞书开放平台后台填写清单.md` 配置应用后台。
- 机器人能力保持开启，机器人已加入测试群。
- 事件订阅选择：

```text
使用长连接接收事件
```

- 至少订阅：

```text
im.message.receive_v1
```

- 启动本地 API 和长连接桥接：

```powershell
npm run dev
npm run dev:lark-events
```

- 机器人自定义菜单先使用“发送文字消息”复用文本触发逻辑：

```text
开始整理 -> /agent 请整理当前群聊讨论，生成需求文档、5页汇报PPT和3分钟讲稿。
查看进度 -> 进度
取消任务 -> 取消
```

- 只处理测试群、文本消息和包含触发关键词的消息。

成功标准：

- 在飞书测试群发送 `@Agent` 或 `/agent` 指令后，`npm run dev:lark-events` 能收到真实事件并转发到本地 API。
- 事件能创建任务，并在 Web / 飞书内嵌页面看到 `planning` 或 `waiting_user` 状态。
- 普通群消息不会触发 Agent。
- 点击机器人菜单“开始整理”能触发同一条主流程。

验证方式：

```powershell
npm run typecheck
npm run build
npm run test:e2e
lark-cli doctor
```

手动验证：

- 飞书后台长连接验证通过。
- 测试群发送一条触发消息。
- `dev:lark-events` 日志能看到事件进入和转发。
- 任务列表出现对应任务。

风险：

- 本地代理可能影响 `lark-cli` 连接飞书；当前已要求 lark-cli 尽量不走聊天代理。
- 飞书后台权限、事件、菜单修改后需要发布版本，客户端侧可能有几分钟延迟。
- 如果真实长连接 payload 和本地模拟 payload 不一致，需要优先记录真实 payload，再扩展 parser。

## 3. 任务二：安全校验、幂等和机器人自消息过滤

目标：

把飞书事件入口从“可用”提升到“演示时不乱触发、不重复执行、不自我循环”。

最小实现：

- 增加飞书事件签名校验或 token 校验。
- 根据 `message_id` 做持久化幂等，而不是只放内存。
- 过滤机器人自己或当前应用自己发送的消息，避免 Agent 回发又触发 Agent。
- 限制只处理白名单测试群。
- 对无法识别的事件返回明确 ignored reason。

成功标准：

- 同一个 `message_id` 重试多次，只创建一个任务。
- Agent 回发的交付消息不会再次触发新任务。
- 非白名单群消息不会触发。
- 签名错误或 token 错误的请求会被拒绝。

建议文件：

```text
apps/api/src/triggers/larkImTrigger.ts
apps/api/src/index.ts
apps/api/src/security/larkEventAuth.ts
apps/api/src/state/TaskStore.ts
```

## 4. 任务三：任务持久化与会话恢复

目标：

避免服务重启后任务、事件、产物和幂等记录全部丢失。

最小实现：

- 引入轻量 SQLite 或本地 JSONL 存储。
- 持久化 Task、AgentEvent、Artifact、handled message IDs。
- 服务启动时恢复最近任务列表。
- 支持按 `chatId` 找到最近一个 `waiting_user` 任务。

成功标准：

- 创建任务后重启 API 服务，任务仍能在 `/api/tasks` 中看到。
- 重启后重复的飞书 `message_id` 不会再次生成任务。
- 重启后用户仍可对 `waiting_user` 任务回复“确认 / 继续”。

暂不做：

- 多租户数据库。
- 云端数据库部署。
- 完整审计日志后台。

## 5. 任务四：群内确认状态机升级

目标：

把当前最小版“确认 / 继续”扩展为更完整的 Agent 会话能力。

当前已有：

```text
IM 触发
  -> waiting_user
  -> 用户回复“确认 / 继续”
  -> Agent 执行
```

第三周期要补：

- `确认`：继续执行。
- `修改计划：xxx`：重新规划或调整步骤。
- `取消`：终止当前任务。
- `进度`：回发当前任务状态和已完成步骤。
- `重新生成 PPT`：只重新生成 Slides，不重做 Docs。

成功标准：

- 用户可以在飞书群内完成至少三类控制：确认、取消、查进度。
- 修改类指令不会破坏已完成产物。
- Web / 飞书内嵌页面能同步显示等待确认、已确认、已取消、执行中、完成等状态。

## 6. 任务五：飞书内嵌页面与多端演示路径

目标：

把当前 Web 仪表盘从开发期控制台包装成飞书应用内可打开的页面。

最小实现：

- 保留 React + Vite 前端。
- 优化移动端和桌面端响应式布局。
- 在飞书应用后台配置网页应用入口。
- 页面中突出三类信息：
  - 当前 Agent 任务状态。
  - 计划步骤与执行日志。
  - Docs / Slides / 讲稿 / 交付摘要链接。
- 支持从 URL query 或本地状态定位最近任务。

成功标准：

- 在飞书客户端中能打开仪表盘页面。
- 移动端视图不溢出、不重叠。
- 桌面端能清晰展示任务步骤、产物链接和确认节点。
- 群聊触发任务后，页面能实时更新。

设计要求：

- 保持办公工具气质，克制、清晰、可扫描。
- 不做营销式 landing page。
- GUI 是辅助仪表盘，不抢 Agent 主流程。

## 7. 任务六：Slides 质量与汇报材料增强

目标：

让真实飞书 Slides 从“有内容”提升到“能用于比赛演示”。

最小实现：

- 优化 `SlidesXmlBuilder` 的页面模板。
- 支持封面、痛点页、方案页、流程页、价值页、下一步页。
- 统一标题、正文、要点和背景样式。
- 为每页增加更稳定的文本布局，避免内容越界。
- 将 3 分钟讲稿写入单独 Docs 或交付摘要，方便演示时打开。

成功标准：

- 生成的 Slides 打开后每页都有可见内容。
- 标题和要点不明显重叠。
- 至少 5 页结构完整。
- 可用作比赛演示稿的初稿，而不是只作为技术验证。

暂不做：

- 完整 PPT 设计器。
- 大量图片和复杂图表自动生成。
- 多主题模板市场。

## 8. 任务七：验证体系与演示脚本

目标：

让第三周期成果可重复演示、可解释、可提交。

自动化验证：

```powershell
npm run typecheck
npm run build
npm run test:e2e
```

新增测试建议：

- 飞书事件签名校验测试。
- 机器人自消息过滤测试。
- 持久化恢复测试。
- 群内确认 / 取消 / 查进度测试。
- Slides XML Builder 单元测试或快照测试。

手动验收脚本：

```text
1. 在飞书测试群发送需求讨论。
2. 发送 @Agent 指令。
3. 群里收到 Agent 规划和确认请求。

```

## 10. 第三周期验收标准

本周期结束时，至少应该能稳定演示：

```text
飞书测试群 @Agent
  -> 真实飞书事件经长连接进入本地 Agent
  -> Agent 读取上下文并规划
  -> 群里收到确认请求
  -> 用户回复确认
  -> Agent 创建真实 Docs + Slides
  -> 飞书内嵌页面实时展示任务状态
  -> 群里收到最终交付链接
  -> Docs 和 Slides 打开后内容可见
```

最低通过标准：

- 不依赖手动调用本地 API 触发任务。
- 不出现重复任务或自我触发循环。
- 不强依赖公网 HTTPS；本地演示优先使用飞书官方长连接。
- 任务失败时有可读错误，而不是静默卡住。
- Web / 飞书内嵌页面和群聊里的状态基本一致。
- 所有真实密钥和真实资源 ID 不进入 Git。

## 11. 不做清单

第三周期暂不做：

- 完整离线 CRDT 合并。
- 原生 iOS / Android 客户端。
- Electron / Tauri 桌面端打包。
- 多租户权限系统。
- 完整白板 / 自由画布编辑器。
- 大规模富媒体自动排版。

理由：

这些能力对最终作品有价值，但第三周期最关键的是把真实飞书事件触发、内嵌页面、任务持久化和演示稳定性做好。只有这条主链路足够可靠，后续加分项才有实际承载基础。

同时，公网 HTTPS webhook 暂不作为第三周期 P0。原因是飞书后台已经支持长连接接收事件，当前比赛开发阶段更应该先把飞书内真实触发、群内确认、真实 Docs / Slides 交付跑顺。

## 12. 阶段小结素材记录要求

第三周期开发过程中每天记录：

- 是否完成一次真实飞书群内触发。
- 是否创建真实 Docs / Slides。
- 事件订阅、签名校验、持久化、内嵌页面分别推进到哪一步。
- 遇到的真实平台问题和解决方式。
- 自动化测试运行次数和结果。
- 演示截图或关键链接是否已留存。

第三周期小结建议指标：

| 指标 | 记录方式 |
| --- | --- |
| 真实飞书事件触发次数 | 群内 @Agent 成功进入后端次数 |
| 确认状态机使用次数 | 确认 / 取消 / 查进度等指令次数 |
| 真实 Docs / Slides 数量 | 成功生成并可打开的产物数量 |
| 持久化恢复验证次数 | 重启后任务和幂等记录是否保留 |
| 飞书内嵌页面验证 | 移动端 / 桌面端打开结果 |
| 自动化验证次数 | typecheck / build / E2E |

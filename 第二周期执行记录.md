# 第二周期执行记录

更新时间：2026.04.28

## 1. 本轮目标

按照第二周期规划，推进真实飞书小闭环的基础能力。在确认测试群后，执行真实群消息读取、飞书 Docs 创建、飞书 Slides 创建、群聊回发和文档追加修改。

本轮成功标准：

- Mock 主链路继续稳定。
- Planner 对豆包输出具备 Schema 校验和重试能力。
- `LarkCliAdapter` 能读取测试群并创建真实飞书 Docs / Slides。
- Web 仪表盘能展示真实 / Mock 运行状态和产物链接。
- 明确记录 Web 仪表盘与最终飞书内嵌形态的边界。

## 2. 已完成

### 2.1 Planner 稳定化

新增：

```text
apps/api/src/agent/schemas.ts
```

完成：

- 使用 Zod 校验 Planner 输出。
- 限制 `steps[].tool` 只能是允许的工具枚举。
- 当豆包第一次输出不符合结构时，自动重试一次。
- 重试仍失败时返回可读错误，避免后续步骤吃到脏计划。

### 2.2 LarkCliAdapter 改进

完成：

- `readMessages` 支持通过 `LARK_DEFAULT_CHAT_ID` 读取默认测试群。
- 飞书消息体会尝试解析 JSON 文本内容。
- `createDoc` 使用临时 Markdown 文件调用 `lark-cli docs +create`。
- `createDoc` 会尝试从 CLI JSON 或文本输出中提取真实文档链接。
- 新增 `sendMessage` 接口，为后续向测试群回发交付摘要做准备。
- 没有配置 `chat_id` 时，会给出明确错误，不静默失败。
- 根据真实 CLI 约束修正 `docs +create` 参数：不再传不存在的 `--format`。
- 根据 Windows 下 `lark-cli` 的路径要求，`--markdown @file` 改为传当前目录内相对路径。
- Windows 下不再通过 shell 拼接 CLI 参数，避免 Markdown 空格被拆成位置参数。
- 兼容 `lark-cli im +chat-messages-list --format json` 的真实结构 `data.messages`。
- 过滤已删除消息和空内容，避免把系统/无效消息算入需求讨论。
- `updateDoc` 已接入 `lark-cli docs +update --mode overwrite`，支持自然语言追加修改真实飞书文档。
- `createSlides` 已接入 `lark-cli slides +create`，将 Markdown PPT 大纲转换为简版 Slides XML。
- 群回发改为纯文本摘要，避免飞书 Markdown 链接触发富文本 URL 校验失败。
- 修复 Slides 空白页问题：飞书 Slides 不能直接写 `<slide><text>...</text></slide>`，必须使用 SML 2.0 结构，即 `<data><shape type="text"><content><p>...</p></content></shape></data>`。
- 新增 `SlidesXmlBuilder`，将 PPT Markdown 大纲转换为可渲染的封面页和内容页 XML。

### 2.3 Agent 交付回发入口

完成：

- `summary.deliver` 阶段会在 Adapter 支持时尝试发送交付摘要。
- 回发失败不会导致主任务失败，而是记录 `integration.warning` 事件。
- 当前最新验证中群回发已成功，测试群能看到 `Agent-Pilot 任务交付` 消息。
- 回发内容包含真实 Docs 和 Slides 链接。

### 2.4 Web 仪表盘增强

完成：

- 顶部显示测试群是否已配置。
- 产物卡片支持打开真实链接。
- 事件列表可以显示 warning / error message。
- Mock E2E 不受本地 `.env` 中真实飞书配置影响。

结果：

- TypeScript 检查通过。
- 生产构建通过。
- Playwright E2E 通过。
- `lark-cli doctor` 通过，当前 token 可用。
- 真实飞书任务可完成：测试群读取 -> 豆包规划 -> 飞书 Docs 创建 -> 飞书 Slides 创建 -> 群回发 -> Web 产物链接展示。
- 真实读群已从 0 条修复为 10 条有效讨论消息。
- 自然语言追加修改已验证：飞书 Docs 文档版本从 v1 更新到 v2。
- 最新端到端任务 `warningCount=0`。
- 已创建真实 Slides 测试文稿并用 `slides xml_presentations get` 读回确认：页面 XML 中包含可渲染的文本框、标题、要点列表和背景样式。

- 确认消息仍走同一个飞书 IM webhook，通过 `chatId` 找到同群最近一个 `waiting_user` 任务，避免把 GUI 当成唯一控制入口。

## 4. 未完成 / 阻塞

真实飞书小闭环已经跑通，当前剩余的是产品化和演示形态问题：

- Web 仪表盘仍是开发期控制台，不是最终嵌入飞书的产物。
- 飞书群内触发已有后端接口，但还没有完成公网 HTTPS 部署、飞书后台事件订阅、签名校验和机器人自身消息过滤。
- 群内确认状态机仍是最小版：支持确认继续，不支持取消、改计划、多人审批、超时恢复和持久化恢复。
- Slides XML 已能渲染文本内容，但视觉和版式还只是简版，缺少正式 PPT 的图表、流程图、图片和统一主题模板。
- 讲稿仍作为 Web Artifact 展示，后续可写入飞书文档或回发到群。

5. 截图记录：群聊触发、Web 任务进度、飞书 Docs、飞书 Slides、群聊回发消息。

验收时不要提交真实 `chat_id`、文档 token 或密钥到 Git。

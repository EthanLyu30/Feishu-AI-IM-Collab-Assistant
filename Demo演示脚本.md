# Demo 演示脚本

## 1. 演示目标

本 Demo 需要清晰展示：

- 用户从 IM 入口用自然语言触发任务。
- Agent 自动理解意图、拆解计划并调用多个场景模块。
- 系统串联 IM、文档、演示稿或自由画布。
- 移动端与桌面端状态实时同步。
- GUI 只作为仪表盘和辅助操作台，核心流程由 Agent 驱动。

## 2. Demo 主线

主线任务：

> 从一段 IM 讨论出发，自动生成需求文档，再生成汇报 PPT，并完成总结交付。

示例业务主题：

> 校园活动报名系统。

选择这个主题的原因：

- 容易理解，不需要复杂业务背景。
- 很适合展示需求提取、功能拆解和演示稿生成。
- 可自然覆盖用户、流程、权限、数据统计和风险等内容。

## 3. 演示角色

- 用户 A：项目负责人，通过 IM 发起任务。
- 用户 B：团队成员，参与讨论或提出修改意见。
- Agent-Pilot：智能助手，负责规划和执行。
- 桌面端仪表盘：展示任务状态、文档和演示稿。
- 移动端界面：展示 IM 入口和同步状态。

## 4. 准备数据

### IM 讨论示例

```text
用户 A：我们需要做一个校园活动报名系统，先支持学生查看活动和报名。
用户 B：老师这边需要发布活动，还要能看到报名人数。
用户 A：最好有截止时间，超过时间不能报名。
用户 B：后台要能导出报名名单，后面做活动签到。
用户 A：汇报时重点讲痛点、方案、核心流程和后续规划。
```

### 用户触发指令

```text
帮我把刚才的讨论整理成一份需求文档，并生成一份 5 页汇报 PPT，最后给我一段 3 分钟讲稿。
```

## 5. 演示步骤

### Step 1：IM 入口触发

移动端展示 IM 对话。

用户输入：

```text
帮我把刚才的讨论整理成一份需求文档，并生成一份 5 页汇报 PPT。
```

Agent 回复：

```text
收到。我会先读取当前会话讨论内容，提取需求点，然后生成需求文档和汇报 PPT。开始前我会先给出任务计划。
```

展示重点：

- 入口是 IM，而不是传统表单。
- 用户只发自然语言指令。
- Agent 主动承接任务。

### Step 2：Agent 生成任务计划

桌面端仪表盘同步出现任务卡片。

任务计划示例：

```text
目标：生成校园活动报名系统需求文档和 5 页汇报 PPT。

步骤：
1. 读取 IM 讨论上下文。
2. 提取业务背景、用户角色和核心需求。
3. 生成需求文档初稿。
4. 根据需求文档生成 PPT 大纲。
5. 生成 5 页演示稿。
6. 生成 3 分钟汇报讲稿。
7. 输出最终交付摘要。
```

展示重点：

- Agent 不是黑盒生成，而是先规划。
- 用户能看到执行链路。
- 移动端和桌面端状态一致。

### Step 3：生成需求文档

桌面端展示文档生成进度。

文档结构示例：

```text
# 校园活动报名系统需求文档

## 1. 背景与痛点
当前校园活动报名依赖群消息和人工表格收集，信息分散、统计困难、截止控制弱。

## 2. 目标用户
- 学生：浏览活动、提交报名、查看报名状态。
- 老师：发布活动、查看报名情况、导出名单。
- 管理员：维护活动分类、权限和数据。

## 3. 核心功能
- 活动发布
- 活动列表
- 在线报名
- 截止时间控制
- 报名名单导出
- 活动签到扩展

## 4. 约束与风险
- 报名重复提交
- 截止时间边界
- 权限隔离
- 数据导出安全
```

展示重点：

- 文档来自 IM 讨论，而不是人工复制整理。
- Agent 自动结构化非结构化对话。

### Step 4：自然语言修改文档

用户在移动端继续输入：

```text
把权限管理补充得更详细一点，并加上学生和老师的不同操作边界。
```

Agent 回复：

```text
我会补充权限管理章节，并同步更新后续 PPT 中的方案页。
```

桌面端展示文档更新。

新增内容示例：

```text
## 权限管理

学生只能查看已发布活动、提交或取消自己的报名。
老师可以创建活动、编辑自己发布的活动、查看报名统计和导出名单。
管理员可以管理全部活动、用户权限和系统配置。
```

展示重点：

- 用户仍然通过自然语言操作。
- Agent 能把修改影响同步到后续产物。

### Step 5：生成 PPT

Agent 将需求文档转成 5 页 PPT。

PPT 结构示例：

```text
第 1 页：校园活动报名系统
第 2 页：现状痛点
第 3 页：解决方案与核心流程
第 4 页：功能模块与权限设计
第 5 页：交付计划与后续扩展
```

展示重点：

- PPT 不是孤立生成，而是由文档沉淀结果转化而来。
- 覆盖 IM、Doc、PPT 三个办公组件。

### Step 6：排练与优化

用户输入：

```text
帮我生成一段 3 分钟讲稿，并指出这份 PPT 哪一页最需要加强。
```

Agent 输出：

```text
讲稿已生成。当前最需要加强的是第 3 页，因为方案流程需要更直观地体现从活动发布到报名统计的闭环。
建议加入一张流程图：老师发布活动 -> 学生报名 -> 系统截止校验 -> 老师导出名单 -> 活动签到。
```

展示重点：

- Agent 不只是生成内容，也能进行质量反馈。
- 体现高级 Agent 能力。

### Step 7：交付归档

Agent 输出：

```text
本次任务已完成：
- 需求文档：校园活动报名系统需求文档
- 演示稿：校园活动报名系统汇报 PPT
- 汇报讲稿：3 分钟版本
- 后续待办：补充签到流程、确认导出字段、设计权限校验细节
```

展示重点：

- 形成正式交付闭环。
- 方便汇报和归档。

## 6. 多端协同展示点

必须在演示中刻意展示以下动作：

- 移动端 IM 发起任务，桌面端立即出现任务。
- 桌面端查看 Agent 执行进度，移动端同步收到状态。
- 移动端发出修改指令，桌面端文档内容更新。
- 桌面端完成 PPT 后，移动端收到交付结果。

## 7. 备用演示方案

如果飞书 API 暂时不可用，可以使用 Mock IM 会话和本地仪表盘演示 Agent 工作流。

备用链路：

```text
Mock IM 消息 -> Agent Planner -> Mock Docs -> Mock Slides -> Sync Dashboard
```

解释口径：

> 当前阶段飞书 API 权限尚未开放，因此先通过 Adapter 层模拟真实 IM、Docs 和 Slides 行为。后续 API 开放后只需要替换 Adapter 实现，Agent 编排、数据模型和多端同步机制可以复用。

## 8. 成功标准

本 Demo 成功的判断标准：

- 观众能看懂 Agent 如何从 IM 对话启动任务。
- 观众能看到清晰的任务规划过程。
- 观众能看到文档和 PPT 是被 Agent 串联生成的。
- 观众能看到移动端和桌面端的同步。
- 观众能感受到 GUI 是辅助台，而不是主流程。

## 9. 可录屏片段

- 30 秒：需求文档生成。
- 20 秒：自然语言修改文档。
- 30 秒：PPT 生成和讲稿生成。
- 15 秒：最终归档交付。

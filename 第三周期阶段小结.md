# 第三周期阶段小结

周期：2026.04.29 - 2026.05.01 阶段性小结

## 一、核心产出（必填）

本周期我最满意的成果，是把项目从“Web Demo 能跑通”推进到了“飞书 IM Agent 可演示雏形”。

具体来说，项目现在已经不只是一个前端仪表盘或本地模拟流程，而是形成了比较完整的 Agent-Pilot 主链路：用户可以通过飞书 IM 指令触发任务，Agent 读取上下文、生成计划、等待群内确认，再驱动 Docs、Slides、讲稿和交付摘要生成，并通过 WebSocket 把执行过程同步到仪表盘。这个闭环覆盖了赛题要求中的 IM、Doc、Slides 三类办公组件，也体现了“AI Agent 是主驾驶，GUI 是辅助仪表盘”的设计方向。

本周期的另一个重要产出，是对工程稳定性和演示可信度做了系统补强。项目新增了飞书消息幂等处理、机器人自消息过滤、任务状态持久化、工具执行事件、产物质量校验、Cloudflare Pages 稳定入口和飞书后台填写说明。尤其是 `ArtifactVerifier` 和 `TaskStore` 的加入，让系统不再只是“生成了内容”，而是能解释产物质量、保留执行记录，并在服务重启后恢复任务状态。

我个人觉得这一周期的转折点，是意识到比赛评审看的不是“有没有按钮能点”，而是 Agent 有没有真正主导办公流程。这个认知让我把开发重点从“堆功能”调整到“让 Agent 的计划、执行、确认、交付和恢复都可观察、可解释、可演示”。

## 三、过程复盘与沉淀（必填）

### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

本周期主要完成了四条线。

第一条是飞书 IM 入口。我们把飞书群内自然语言指令接入到 Agent 任务系统，补充了 `确认 / 继续`、`进度`、`取消` 等会话控制指令，并增加同群活跃任务互斥，避免同一个群同时挂多个任务。

第二条是 Agent 执行链路。项目新增了工具级事件，让每个工具调用都能记录开始、完成、耗时和失败原因。这样在仪表盘里看到的不只是任务状态，而是更接近 Hermes-style 的 Tool Loop：Agent 在什么步骤、调用什么工具、花了多久、生成了什么产物，都能被追踪。

第三条是演示材料质量。针对之前飞书 Slides 能打开但内容或版式不够稳定的问题，我把 Slides XML Builder 继续增强，增加了封面、流程页、角色权限页、计划页和总结页的差异化布局，并调整提示词，要求豆包生成 6 页汇报结构、短要点和讲者备注。

第四条是稳定性与文档。项目新增了任务状态持久化、产物质量校验和本地运行体检脚本，也把飞书后台菜单、Cloudflare Pages、Named Tunnel、云服务器选择等配置路径写进文档。开发过程中主要使用 Codex 辅助代码实现、文档整理和测试补充；使用 Playwright 做 E2E 验证；使用 `larksuite/cli` 和 Cloudflare 相关工具验证飞书与部署链路。

### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

最卡的是飞书开放平台的使用方式。一开始我以为申请了 App ID 和 App Secret 后，应用就自然“部署在飞书里”了；后来才理解，飞书开放平台提供的是入口、事件、权限和办公 API，真正的 Agent 后端仍然需要自己运行或部署。这个认知差导致我一度纠结公网 HTTPS、长连接、机器人菜单和网页应用入口之间的关系。

破局方式是把飞书接入拆成两条线：事件触发优先使用飞书官方长连接，仪表盘访问使用 Cloudflare Pages 稳定入口。这样短期不强行切 webhook，也能完成真实飞书群内触发和飞书内页面展示。

另一个坑是机器人菜单。“开始整理 / 查看进度 / 取消任务”占满了三个一级菜单后，“打开仪表盘”不会出现。后来把菜单重排为“任务 / 仪表盘 / 帮助”，或在三菜单模式下用“开始整理 / 打开仪表盘 / 取消任务”，才解决这个入口问题。

Slides 也踩过坑。最开始生成的飞书演示稿能打开，但内容可能为空或视觉很弱。后来通过研究飞书 Slides XML 的文本 shape 结构，把 Markdown 转换成更稳定的 slide XML，并增加产物校验，避免只把“创建成功”当成“内容可用”。

### 3. 有没有什么这次写得特别顺，或者下次还能直接复用的东西？

这次最值得复用的是几个工程边界。

第一是 Adapter 边界。`MockOfficeToolAdapter` 和 `LarkCliAdapter` 让 Agent 主流程不被具体飞书实现绑死，后续如果改成飞书 OpenAPI SDK，也可以替换 Adapter，而不是重写整个 Agent。

第二是事件与状态模型。Task、Step、Artifact、AgentEvent 这套结构可以继续支撑后面的多端协同、任务恢复、演示录屏和阶段小结素材积累。

第三是飞书后台配置清单。现在已经沉淀了机器人菜单、长连接、权限、网页应用、Cloudflare Pages、Named Tunnel 等填写策略，后续切换服务器或正式部署时可以直接复用。

第四是“生成后校验”的开发习惯。之前只关心 LLM 是否输出、API 是否成功；现在会用 Verifier 检查文档结构、Slides 页数、讲者备注和交付摘要完整度。这套思路后续可以扩展到富媒体、归档、权限边界和演示脚本校验。

## 四、本周随手记（选填）

这一周期最大的感受是，Agent 项目不能只看“模型会不会生成文本”。真正难的是让 Agent 在一个复杂办公环境里可控地行动：什么时候读群消息，什么时候问用户确认，什么时候创建文档，什么时候生成 PPT，失败了怎么解释，重复事件来了怎么避免误触发。

我也更能理解赛题里“Pilot / Co-pilot”的意思。GUI 不是越复杂越好，它更像飞行仪表：把 Agent 的计划、进度、风险和交付物清楚展示出来，让用户信任 Agent 正在正确地推进任务。

下一步我准备先向赛事官方询问能否申请火山引擎服务器。如果能拿到官方服务器，就优先把 API 和飞书长连接放到云端常驻；如果暂时拿不到，就继续保持本地长连接 + Cloudflare Pages 的演示路线，同时重点打磨 Agent 执行质感、Slides 汇报质量和飞书内嵌体验。

这周也有一点明显的心态变化：一开始我更在意“功能有没有接上”，现在会更在意“评委看到时是否觉得这是一个真实可用的协同助手”。这个标准更高，但也更接近这个题目真正想考察的东西。

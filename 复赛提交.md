### 2.4 AI 亮点介绍

#### 2.4.1 项目中使用的高阶 AI 技巧

- **意图主动澄清（Clarifier）**：豆包模式下检查用户意图是否需要先澄清，避免对模糊请求"硬上"生成；用户补充后不会反复追问，已用 E2E 锁定。
- **可解释计划生成（Planner）**：豆包输出步骤化执行计划，附 JSON Schema 校验与失败重试；每个工具调用记录 `tool.started / tool.completed / tool.failed` 三类事件，仪表盘可时序回放。
- **产物自动校验（Verifier）**：Docs 字数、章节数、权限/角色关键词；Slides 文本长度、版式覆盖、讲者备注完整度。校验失败会进入"失败可重试"路径而不是静默通过。
- **Slides 模板自动选型**：标题 / 正文关键字命中"流程 / 角色 / 计划 / 数据 / 对比 / 总结"等模式时自动切换到对应版式（process / roles / timeline / stats / comparison / summary），减少"AI 生成的 PPT 永远长一个样"。
- **失败恢复建议**：任务失败时记录失败步骤、工具、错误原因和恢复建议，仪表盘单点重试。
- **有界 LLM 调用 + 稳定回退**：所有豆包调用统一 timeout + fallback，避免单次抖动拖垮整链路。
- **外部依赖降级**：飞书群历史读取失败、机器人 chat scope 缺失等场景，Agent 仍能继续推进可控部分。

#### 2.4.2 人和 AI 的分工

- **AI 负责**：意图理解、信息补全提问、计划生成、Docs/Slides 内容生成、产物自检、失败恢复建议、阶段总结撰写素材。
- **人负责**：业务边界与权限策略、Slides XML 模板与版式自动选型规则、稳定性约束与降级路径、飞书后台与服务器运维、最终评审视角的审美与节奏。
- **共建**：当 AI 输出不稳定时，人补一层"边界 Schema + Verifier"；当人陷入主观调试循环时，引入 AI 的方法论 skill（如 `frontend-design`）强行换框架。

#### 2.4.3 核心模型选型思路

- **主模型**：豆包 2.0 Pro（火山方舟），通过 OpenAI 兼容协议接入。理由：中文办公场景表达力强、上下文容量充足、工具调用与 JSON 模式稳定、与赛题方向一致。
- **接入方式**：OpenAI 兼容 SDK，便于快速切换到任意 OpenAI-compatible endpoint；Mock 模式（`AGENT_LLM_MODE=mock`）保证 E2E 与离线开发可运行。
- **未来拓展**：保留 LLM Adapter 边界，可平滑接入更长上下文、专家小模型或在端侧蒸馏的本地模型。

#### 2.4.4 引入 AI 后对原有工作流的改变

- **从"人手动整理 IM → 写 Docs → 做 PPT → 回发链接"** 缩短为 **"群内一句话 → Agent 自动澄清 + 规划 + 生成 + 校验 + 回发"**。
- **从"成员盯进度"** 升级为 **"运行台同步任务时间线 + 工具耗时 + 产物校验 + 失败重试"**。
- **从"演示靠人讲解"** 升级为 **"评委一眼看懂 A-F 场景覆盖 + KPI 卡片 + 前后对比 + 真实可打开产物链接"**。
- **从"AI 出错就停"** 升级为 **"AI 出错局部降级，可控部分继续推进，失败步骤可解释、可重试"**。

### 2.5 其他信息补充

- **Demo 安全边界**：当前事件接入主线为飞书长连接，暂不依赖 webhook 公网回调；如切 webhook 已沉淀 raw body 签名校验文档。
- **持久化**：任务、事件、产物、`messageId` 幂等均通过 `.data/*.json` 本地持久化，适合比赛演示与轻量部署；长期生产建议迁移 SQLite / 云数据库（已列入路线）。
- **多端**：当前移动端与桌面端是同一份 Web 应用在飞书客户端内的两种宽度形态，并非原生客户端。后续若引入 Expo React Native 或 Tauri，整体仍可继续以 TypeScript 为主语言。
- **未完成项（透明声明）**：原生 iOS / Android 客户端、语音输入、富媒体 / 自由画布、Docs 归档到指定文件夹、SQLite 持久化、SSH key 改造。

# 第四周期阶段小结

周期：2026.05.02 - 2026.05.04 阶段性整理

## 一、核心产出（必填）

本周期我最满意的一项成果，是把项目从“本地可演示 / 临时 Tunnel 可访问”升级成了“轻量云服务器 + HTTPS + 飞书长连接 + systemd 常驻”的稳定演示形态。

这一阶段之前，项目虽然已经具备 IM 触发、Agent 编排、Docs / Slides 生成和 Web 仪表盘，但整体仍然比较依赖本机环境：本地服务、临时公网地址、飞书后台 URL、长连接进程都可能成为演示时的不确定因素。本周期完成轻量云服务器部署后，项目有了一个稳定入口：

```text
https://agent-pilot.47-236-122-49.sslip.io
```

这个入口同时服务于飞书机器人菜单、桌面端网页应用和移动端网页应用。后端 API、Web 页面、飞书长连接事件桥接均已在 Ubuntu 服务器上常驻运行，并通过 Nginx、Let's Encrypt 证书和 systemd 完成基础工程化部署。这样一来，项目不再只是“我电脑上能跑”，而是具备了更接近真实应用的在线访问、服务自启动、健康检查和故障排查能力。

本周期的另一个关键成果，是把 Web 仪表盘从早期的演示页面重构为更贴近赛题定位的 Agent 运行台。它现在不只是展示结果，而是围绕“Agent 是主驾驶，GUI 是辅助仪表盘”的思路，展示自然语言任务入口、A-F 场景覆盖、当前 Pipeline、生成产物、飞书 IM 入口预览、运行检查和连接配置。评委打开页面后，可以更直观地看到这个系统不是单一固定链路，而是由 IM、Planner、Docs、Slides、多端同步和交付总结组成的可组合办公 Agent。

我个人觉得这一周期的转折点，是从“能不能接上飞书”转向“能不能被稳定、可信、可复盘地展示”。这让我开始把部署、健康检查、自检脚本、UI 证据和飞书后台配置都当成项目的一部分，而不是额外的杂事。

## 三、过程复盘与沉淀（必填）

### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

本周期主要完成了五个环节。

第一是稳定部署。我们在轻量云服务器上完成了 Node.js、Nginx、Certbot、`larksuite/cli` 和项目依赖的安装，把 Agent API、Web 页面和飞书长连接事件桥接部署到 `/opt/agent-pilot`，并使用 systemd 保证服务重启后自动恢复。相比之前的临时 Tunnel，这条路线更适合比赛答辩和远程检查。

第二是飞书接入方式重新梳理。通过进一步理解飞书开放平台，我明确了飞书并不会托管我们的后端代码，而是提供机器人、事件、权限、菜单和网页应用入口；真正的 Agent 运行时仍然需要自己部署。因此当前采用“飞书长连接接收事件 + 自有 HTTPS 页面展示运行台”的方案。这样既避免了 webhook 公网签名校验在当前阶段带来的额外复杂度，又能保持飞书群聊触发和飞书内打开页面这两个核心体验。

第三是运行台 UI 升级。参考 Harness、Vercel Dashboard 等偏工程化产品的设计思路后，我把页面从普通演示仪表盘改成了更像 Agent Run Console 的结构：顶部展示 LLM、Office、API 状态，左侧是导航，主区展示任务入口、Pipeline 和产物，右侧展示飞书入口预览、追加修改、下一步建议和 readiness。移动端也重新调整了顺序，让自然语言任务入口优先出现。

第四是自检和质量门禁。新增 `doctor:ui` 和 `harness:loop` 后，项目可以自动完成类型检查、构建、本地栈启动、桌面 / 移动端截图、DOM 检查和视觉约束检查。自检规则里包括 A-F 场景是否存在、页面是否横向溢出、主要卡片圆角是否超过 8px、关键文字是否溢出等，这让 UI 不再只靠主观感觉判断。

第五是文档和演示材料沉淀。我把轻量云部署过程、低成本部署路线、飞书后台填写方式、Bot Open ID 获取方式、UI Harness 使用方法和第四周期任务规划都写进项目文档。开发过程中主要使用 Codex 做代码实现、文档整理和自检脚本设计；使用 `larksuite/cli` 验证飞书能力；使用 Playwright 和项目自检脚本验证 UI 与主流程；使用 Git 保留阶段性提交记录。

### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

这一周期最卡的地方，是对“飞书开放平台到底托管什么”的理解。一开始我以为在飞书开放平台创建应用后，应用就会像小程序一样完全运行在飞书里；后来才意识到，飞书提供的是入口和能力，不会自动运行本仓库里的 Node API、豆包调用、任务状态和 `lark-cli` 长连接。这个认知差导致我一度纠结是不是一定要公网 HTTPS、是不是必须购买域名、是不是只能部署到某种官方服务器。

破局方式是把问题拆开：飞书负责“用户从哪里进来、消息怎么触发、机器人怎么展示”，我们的服务器负责“Agent 怎么运行、任务怎么编排、文档和 PPT 怎么生成、状态怎么保存”。在这个拆分下，当前阶段最合适的路线不是立刻大改架构，而是用轻量云服务器承载运行时，用 `sslip.io` 解决低成本域名，用长连接解决事件订阅，用 Nginx 和 HTTPS 解决飞书入口要求。

另一个不顺的是 UI 观感。早期页面虽然能展示流程，但更像内部调试面板，视觉上不够像一个参赛作品。后来我没有继续堆装饰，而是把 UI 的目标重新定为“评委能快速理解系统能力”。于是加入 A-F 场景覆盖条、运行检查、飞书入口预览、自然语言入口和多端响应式布局，并用 Harness 自检把设计要求变成可重复验证的规则。

还有一个坑是飞书机器人自身消息过滤。机器人回发消息后，如果不做过滤，后续可能被自己的消息再次触发任务。最后通过 `npx --yes @larksuite/cli api GET /open-apis/bot/v3/info --as bot --format json` 获取机器人 `open_id`，写入 `LARK_BOT_OPEN_ID`，再通过 readiness 检查确认过滤项通过。

### 3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？

这次最值得复用的是“轻量云服务器 + 免费通配 DNS + HTTPS + systemd”的部署范式。`sslip.io` 可以直接把域名中的 IP 解析到服务器公网地址，短期内不需要购买正式域名，也不需要额外做 DNS 解析，非常适合比赛早期和中期把系统先稳定跑起来。

第二个可复用的是 readiness 驱动的开发方式。现在系统不是只看服务是否启动，而是检查豆包、Office Adapter、默认群、群白名单、机器人自消息过滤、任务状态和幂等状态等关键项。这个方式能把“演示前到底还缺什么”变得很清楚，后续加功能时也不容易破坏主链路。

第三个可复用的是 UI Harness。以前 UI 好不好主要靠截图和主观感受，现在可以用 `doctor:ui` 和 `harness:loop` 自动检查桌面端、移动端、场景覆盖、横向溢出和文字溢出。这个思路后续也可以扩展到 Docs / Slides 质量检查、飞书群聊触发检查和演示证据包生成。

第四个可复用的是飞书后台配置清单。机器人菜单、网页应用桌面端主页、移动端主页、长连接事件、权限、Bot Open ID、群白名单这些配置都已经被拆成清晰步骤，后面换服务器、换域名或正式答辩前复查时，可以直接按清单执行。

## 四、本周随手记（选填）

这一周期让我很明显地感觉到，Agent 项目不是“模型输出内容 + 前端展示一下”就结束了。真正能打动评委的部分，是它能不能在一个真实办公环境里稳定行动：从飞书群里捕捉需求，判断是否需要澄清，规划子任务，生成文档和 PPT，再把结果回发给团队，并且整个过程可以被观察、被恢复、被复盘。

服务器部署这件事一开始让我觉得有点烦，因为它不像生成 PPT 或设计 Agent 流程那么“显眼”。但做完之后反而感觉项目一下子踏实了很多。以前是“只要我的电脑别出问题就能演示”，现在是“这个系统已经有一个稳定入口和常驻运行方式”。这种变化对信心提升很大。

我也意识到，比赛里的“多端协同”不能只靠口头说明。桌面端、移动端、飞书机器人菜单、Web 运行台、群聊回发和 Office 产物要互相对得上，评委才会觉得这是一个完整系统。第四周期做的很多事情，比如统一 URL、移动端布局、A-F 场景条、readiness 检查，其实都是为了让这个完整性更容易被看见。

下一步我希望继续把项目从“稳定可演示”推进到“优秀可答辩”：补一次真实飞书群聊端到端录屏，整理演示证据包，继续提升 Docs / Slides 的内容质量，强化 Agent 澄清和失败恢复能力，并把服务器安全从 root 密码登录逐步改成更规范的 SSH key 和独立部署用户。

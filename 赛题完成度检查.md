# 赛题完成度检查

更新时间：2026.04.27

## 1. 当前技术栈

当前项目主要使用 TypeScript 作为主语言。

已用技术栈：

| 层级 | 当前技术 |
| --- | --- |
| Monorepo | npm workspaces |
| 后端 API | Node.js + Express + TypeScript |
| 实时同步 | WebSocket / `ws` |
| 前端 | React + Vite + TypeScript |
| 共享类型 | TypeScript shared package |
| LLM 接入 | OpenAI SDK 兼容火山方舟 / 豆包 2.0 Pro |
| 飞书接入 | `larksuite/cli`，通过 `LarkCliAdapter` 封装 |
| 测试 | Playwright E2E + TypeScript typecheck + production build |
| 文档 | Markdown |

## 2. TypeScript 是否合适

结论：当前阶段合适，但不能止步于“Web 模拟多端”。

TypeScript 适合当前项目的原因：

- 飞书 CLI、OpenAI-compatible SDK、WebSocket、React 仪表盘都能很好地用 TypeScript 串起来。
- `packages/shared` 可以复用 Task、Plan、Artifact、Event 类型，减少前后端协议漂移。
- 后续如果采用 Expo React Native 做移动端、Electron / Tauri 做桌面端，仍然可以继续以 TypeScript 为主语言。
- 对比赛早期来说，TypeScript 能最快跑通 Agent 编排和办公套件自动化主链路。

不足：

- 当前移动端和桌面端只是同一个 Web 页面中的双视图模拟，不等于真实 iOS / Android + Windows / macOS 多端。
- 如果评审严格考察客户端基础，后续需要至少补一个真实移动端壳或桌面端壳。
- 如果要深做原生能力，例如系统通知、离线缓存、后台同步、语音输入，可能需要接触平台能力。

建议：

第二周期继续 TypeScript，不引入 Swift / Kotlin。先把真实飞书小闭环跑稳。第三周期再根据时间选择：

- 移动端：Expo React Native。
- 桌面端：Electron 或 Tauri。
- 语音输入：优先 Web Speech / 飞书语音消息转文本能力，后续再考虑原生录音。

## 5. 当前最关键缺口

按优先级排序：

1. 真实飞书小闭环还差群消息发送授权。
2. 测试群目前缺少有效业务讨论，真实生成质量还没有演示素材支撑。
3. 豆包 2.0 Pro 已接入，但还需要输出质量 Verifier 和超时/降级策略。
4. Slides 还只是 Markdown 内容，没有真实飞书演示稿。
5. 多端仍是 Web 模拟，没有真实移动端 / 桌面端载体。
6. 缺少持久化存储，任务重启会丢失。
7. 缺少内容 Verifier 和失败降级策略。
8. 语音入口未完成。

## 6. 第二周期应优先完成

第二周期只做一个真实小闭环：

```text
测试群 IM 消息
  -> Agent 读取上下文
  -> 豆包 2.0 Pro 生成需求文档
  -> 创建真实飞书 Docs
  -> 回发文档链接到测试群
  -> Web 仪表盘同步展示
```

完成这条链路后，项目就具备“AI Agent 驱动飞书办公套件”的真实演示基础。

## 7. 暂时不建议做

第二周期前半段不建议做：

- 原生 iOS / Android。
- Electron / Tauri 打包。
- 完整 PPT 设计系统。
- 离线 CRDT。
- 富媒体插入。
- 多租户权限系统。

理由：

这些功能都重要，但会分散当前最关键的真实 IM + Doc Agent 闭环。先把一条真实链路做稳，再扩展。

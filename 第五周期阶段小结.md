# 第五周期阶段小结

周期：2026.05.05 - 2026.05.07 阶段性整理

## 一、核心产出（必填）

本周期我最满意的一项成果，是在第四周期"稳定可演示"的基础上，把 Agent-Pilot 推进到了"经得起评委拿放大镜看"的复赛形态。

第四周期我们已经有了真实的飞书长连接、稳定 HTTPS 入口、systemd 常驻服务和首版 Agent 运行台。但当我反复打开自己的页面、模拟评委视角看时，它仍然有几个明显的"参赛感"问题：Slides 模板版式偏少，缺少能直接打动人的关键数字与对比；运行台的视觉是一种"工具人"调色板，长时间看会觉得拙；侧边栏几个导航按钮其实是哑按钮，点了没反应；以及，一旦飞书长连接读群消息失败、豆包响应抖动或机器人没分到 chat 范围，整个 Agent 链路就可能被卡死。

本周期围绕这几个真实痛点做了一轮闭环升级：

- **Slides 表达力**。Slides XML Builder 从 7 种版式扩展到 9 种，新增 KPI 卡片（stats）和前后对比（comparison）两种"评委一眼能记住"的版式，并按标题/正文关键字自动选型。覆盖 E2E 同步从 16 条增加到 18 条。
- **复赛叙事抓手**。新增 `项目架构与评分映射.md`，把 mermaid 端到端架构图、核心模块速览和评分三维度（完整性/创新性/技术实现性）的证据矩阵全部沉淀进仓库，给复赛文档和现场答辩准备一份开箱即用的素材。
- **仪表盘视觉重做**。把仪表盘从冷蓝灰 + Inter 的"AI slop"色板，重做成 "Mission Control Editorial" 风格：暖米色羊皮纸工作台、深海军侧边栏配锈红强调条、Instrument Serif 标题、IBM Plex Sans 正文、JetBrains Mono 数字、纸面 grain 噪声 + 入场 staggered fade-in。同时把侧边栏 4 个导航按钮接入视图状态，运行台/流程/产物/检查 4 种聚焦视图各负其责。
- **Agent 鲁棒性补强**。三次连续 fix：飞书读群消息失败时 Agent 仍能继续走规划与生成，不再整链路阻塞；豆包/澄清/规划环节统一加上有界超时与稳定回退；机器人尚未拿到 chat 范围权限时 Slides 也能拿到草稿继续往下走。

我个人觉得这一周期的转折点，是开始用"评委不是开发者"的视角检视项目。第四周期之前更多在补"它能不能跑"，本周期开始问"它好不好看、好不好讲、出问题时是否能优雅降级"。这三个问题驱动了本周期几乎所有改动。

## 三、过程复盘与沉淀（必填）

### 1. 这周主要搞定了哪些具体环节？用了什么方法或工具辅助？

本周期主要完成了四条线。

第一条是 **Slides 表达力升级**。我先把第四周期已经写好的 SlidesXmlBuilder 重新审视了一遍，识别出"它现在只能讲故事，不能给数据"。于是新增了 stats 版式（最多 4 张 KPI 卡片，自动从行内提取数字 + 单位 + 标签，例如把"82-88 分：复赛预估"切成卡片）和 comparison 版式（识别 `→ / -> / vs / 升级为` 等模式生成 Before/After 双栏）。两种版式都补了 E2E，确保"如果 Markdown 里包含数据或对比，至少会被渲染成专门的 slide 而不是退化成 bullet"。

第二条是 **复赛叙事素材**。开始写 `项目架构与评分映射.md` 时我发现，自己一直把架构和评分维度藏在脑子里，没有沉淀成一份能直接被引用的图文。于是用 mermaid 画了端到端 flowchart（IM → 长连接 → API → Planner → 豆包/Office → 回发 / 仪表盘），把核心模块和路径整理成表，再按完整性/创新性/技术实现性三个维度逐项列出"评委关注点 + 项目证据"。复赛作品文档可以直接引用这份素材。

第三条是 **仪表盘视觉重做**。这一步最关键的是先承认"现在的 UI 是 AI slop 配色"，再正式调用 Claude Code 的 `frontend-design` skill，按它推崇的"提交一个明确的美学方向"原则给自己定了一个具体风格："Mission Control Editorial"——暖米色羊皮纸工作台 + 深海军侧边栏 + 锈红强调条 + 编辑级 Instrument Serif 标题 + 技术感 IBM Plex Sans 正文 + JetBrains Mono 数字。这样不会再陷入"再改一点点感觉好像也没差"的循环。同时，把侧边栏 4 个导航按钮真正接进 view 状态：运行台展示全景、流程聚焦 Pipeline、产物聚焦 Artifacts、检查聚焦 Readiness + Endpoint。点击后顶部 kicker 还会同步显示当前视图名。重构完后用 Playwright MCP 真机打开浏览器、截图三种视图，确认风格生效、按钮可切换。

第四条是 **Agent 鲁棒性补强**。三次连续 fix 都是为了"演示中途出小问题不要整体崩"：当读取飞书群历史消息失败时，AgentOrchestrator 会用上下文回退继续规划，不再整任务失败；Clarifier、ContentComposer、Planner 三个调用豆包的关键路径都补了有界超时和稳定回退；当机器人尚未被授予某个群的 read 权限时，Slides 草稿也会继续生成，先保留产物再让用户决定是否回发。

工具与分工上：用 Claude Code 做主驾驶（代码生成 + 文档撰写 + Playwright MCP 截图验证 + 自动 memory），用 `frontend-design` skill 把视觉决策从"主观"变成"按方法论"，用 `npm run doctor:delivery` 一键体检（typecheck / build / e2e / 线上 readiness），用 git 小步提交保留每一步可追踪的差异。

### 2. 过程中有没有遇到什么特别不顺、卡住很久的情况？后来是怎么破局的？

这一周期最卡的是 **"UI 怎么改才不像普通 AI Demo"**。第四周期已经把布局、A-F 场景条、运行检查这些功能性要素补齐了，但视觉上仍然是"白底 + 蓝灰边框 + Inter"的常见风格，看久了就觉得拙。一开始我尝试微调颜色和圆角，越改越像在贴创可贴。

破局是从两个动作开始：

第一是 **强行换框架而不是微调**。调用 `frontend-design` skill 后，它的核心提示是"承诺一个明确而强的美学方向"，不要在多个方向之间摇摆。我据此给自己定了"Mission Control Editorial"——既不走纯 minimalism，也不堆 maximalism，而是沿着"工程控制台 + 编辑级排版"这条具体线路同时改色板、字体、栅格、强调色、入场动画、噪声纹理，一次性提交一个统一方案。

第二是 **用 Playwright MCP 即时验证**。改完 CSS + App.tsx 后，立刻 `npm run dev:e2e` 启动栈，用 Playwright MCP 真机访问 `http://localhost:15173/`，先截一张 1480 × 920 的全景图看整体气质，再依次点击侧边栏 4 个按钮截屏，确认视图切换 + 顶部 kicker 同步 + 锈红激活条都到位。这种"改完立刻看"的循环，比凭感觉调要快很多。

另一个不顺的是 **Agent 在外部依赖抖动时容易整体停摆**。一开始的 AgentOrchestrator 假设豆包会按时返回、飞书会授权所有需要的群 read 权限。但实际演示前自检发现：豆包偶尔会卡几秒、飞书测试机器人没有被加入某些子群、群历史消息读取偶发失败。一旦其中任意一个卡住，整个任务链就停在那里。

破局方式是把"假设外部依赖正确"改成"为每一步外部调用准备一个降级路径"：豆包调用统一加超时 + 稳定 fallback、群历史读取失败时用本地上下文继续规划、机器人没有 chat scope 时 Slides 草稿仍然生成。三次小 fix 加在一起，让 Agent 在演示当天即使遇到飞书侧偶发抖动，也能给出"局部降级 + 继续推进"的体验，而不是整链路黑屏。

还有一个小坑是 **侧边栏按钮看似是按钮但其实是死的**。第四周期我们在侧边栏画了运行台/流程/产物/检查 4 个 NavItem，但代码里只是 `<button>` 没有 `onClick`。这种"长得像可以点其实点不动"的 UI 反而比没有按钮更糟，因为用户会反复点。本周期用 React `view` 状态彻底接通：4 个按钮按 `view` 切换主区域显示哪些 Panel，默认运行台保持全景以兼容 E2E 选择器，不破坏既有的 Playwright 用例。

### 3. 有没有什么你觉得这次写得特别顺，或者下次还能直接复用的东西？

这次最值得复用的有四件事。

第一是 **"先定美学方向，再写 CSS"** 的范式。`frontend-design` skill 给的最直接价值，是把"先承诺一个明确风格"作为开工前的硬约束。后续无论是再做一个内嵌移动端模板、做一份对外营销页、还是给老师做汇报模板，都可以套这个流程：选 1 个具体形容词组（如"Editorial 控制台"、"Brutalist Logbook"、"Soft Pastoral Knit"），先决定字体三件套和色板，再开始写组件。

第二是 **mermaid 架构图 + 评分维度证据矩阵**。这套结构稍作改动就可以变成任何参赛/答辩文档的"项目页"模板：架构图建立心智，模块速览建立工程信赖，评分映射表替你回答评委没问出口的问题。后续做毕设、做面试 portfolio 都能用。

第三是 **"为外部依赖准备 fallback"** 的开发动作清单。这一周期补 3 个 fix 时形成的判断逻辑很清晰：(1) 任何 LLM 调用必须有 timeout + fallback 文本；(2) 任何飞书 API 调用必须能识别"权限范围不足"而不是把任何 4xx/5xx 都视为致命；(3) 任何长链路任务在某一步外部依赖失败时，先尽量推进可控部分，再把不能完成的部分作为可解释的事件抛给运行台。这套清单后续接 webhook 签名校验、归档到指定文件夹、富媒体处理时都可以直接复用。

第四是 **Playwright MCP 即时验证 UI 改动** 的工作流。以前改完 UI 要么靠自己手动开浏览器看，要么靠 E2E 用例间接覆盖。这次直接在 Claude Code 里启动 `dev:e2e`、调用 Playwright MCP `browser_navigate / browser_click / browser_take_screenshot`，把"截图证据"作为变更说明的一部分发回来。这个工作流以后做任何前端改动都可以保留：改完即截图，截图即评审材料。

## 四、项目介绍 + Demo 录屏演示 + 项目链接

### 项目简介

Agent-Pilot 是一个面向飞书办公场景的 AI Agent 协同助手。用户在飞书群里通过 `@Agent` 或 `/agent` 自然语言触发任务后，Agent 会主动澄清模糊指令、规划执行步骤、调用豆包 2.0 Pro 生成需求 Docs 与汇报 Slides、把真实可打开的飞书办公产物链接回发到群里，并通过 Web 运行台同步展示任务时间线、工具耗时、产物校验和失败恢复入口。系统覆盖赛题 A-F 六类场景：IM 入口、任务规划、Docs、Slides、多端协同、总结交付。

技术上是一个 TypeScript monorepo（npm workspaces）：Node.js + Express + WebSocket 后端、React + Vite 前端、`packages/shared` 共享类型；通过 `larksuite/cli` 接入飞书长连接事件、Docs / Slides 真实写入；豆包 2.0 Pro 通过 OpenAI 兼容协议接入；状态以 JSON 文件持久化（任务、事件、产物、消息幂等去重）；线上以 Ubuntu 24.04 + Nginx + Let's Encrypt + systemd 三服务常驻，配 `npm run doctor:delivery` 一键自检。

### Demo 录屏

- **建议演示动线**：飞书测试群发送 `/agent` → Agent 主动澄清并等待用户确认 → 切回 Web 运行台展示任务时间线、工具耗时、Verifier → 打开真实 Docs / Slides（重点演示新增的 KPI 卡片版式与前后对比版式）→ 回到飞书群展示回发的链接、讲稿和下一步建议 → `npm run doctor:delivery` 与线上 `/api/readiness` 截图证明可被远程验收。
- **录屏文件**：第五周期内补录后挂在 `docs/demo/`（占位）；同时保留 Playwright MCP 实拍的运行台 `overview / artifacts / checks` 三视图截图作为静态证据。

## 五、本周随手记（选填）

这一周期开始用 Claude Code 替代 Codex 作为主驾驶，最直观的差异是 skill 体系。`frontend-design` 这种 skill 不是简单的 prompt 模板，而是一套带方法论的执行准则——它会逼我先承诺一个明确的美学方向再开工。这种"工具反过来约束我"的体验，比任意一个单独的 prompt 都更能稳定结果。

另外一件让我有点惊喜的事：自动 memory。我不需要每次都重新告诉 Claude Code "我在做什么"，它会自己读 `MEMORY.md` 拿到我的 user_profile、project_context、feedback_style，再决定怎么开工。这让"自主迭代"从一个口号变成了真实可用的工作模式：我说一句"想法子提升项目质量"，它会自己拉出剩余任务、挑性价比最高的事情做完、跑 typecheck + E2E + delivery doctor、commit + push、再回过头来更新 memory。

下一周期我希望把重心放在三件事：把真实飞书群聊端到端走完一次完整录屏作为复赛硬证据；把任务回放（工具事件时序 + 耗时 + 产物校验）单独做成一个仪表盘视图；把服务器从 root 密码登录改成独立部署用户 + SSH key，让"工程规范、稳定性、可扩展性"这条评分线再往上拱一格。

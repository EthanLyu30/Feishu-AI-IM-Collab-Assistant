# 飞书应用初版交付与部署指南

更新时间：2026.05.01

## 1. 当前交付边界

本轮初版完成的是“飞书内可接入的仪表盘与机器人主链路准备”：

```text
飞书机器人消息
  -> 长连接事件
  -> 本地 Agent API
  -> 豆包 2.0 Pro / 飞书 Docs / 飞书 Slides
  -> Web 仪表盘展示任务状态与产物
```

关键变化：

- Web 仪表盘不再硬编码 `localhost:8787`，可以通过环境变量连接到远程或隧道后的 API。
- 已补 Cloudflare Pages 部署脚本。
- 已补 Cloudflare Tunnel 启动脚本，用于把本地 API 暂时暴露成 HTTPS URL。
- 飞书后台可以先把网页应用入口指向 Cloudflare Pages 页面，把机器人入口继续走长连接。

重要边界：

- 飞书开放平台负责应用入口、机器人、菜单、权限和事件订阅。
- 它不会自动托管仓库里的 API / React 代码。
- 当前最稳妥路线是：前端部署到 Cloudflare Pages，后端 Agent 运行时先保留本地长连接；等演示稳定后，再考虑把后端拆到云端。

## 2. 本地必跑进程

开发和比赛现场演示时，至少开两个终端：

```powershell
npm run dev
```

```powershell
npm run dev:lark-events
```

如果要让 Cloudflare Pages 上的仪表盘访问本地 API，再开第三个终端：

```powershell
npm run tunnel:api:cloudflare
```

该命令会调用 `cloudflared tunnel --url http://localhost:8787`，生成临时 HTTPS 地址。

如果提示没有 `cloudflared`，先安装 Cloudflare Tunnel 客户端：

```text
https://developers.cloudflare.com/cloudflare-one/connections/connect-networks/downloads/
```

## 3. Cloudflare Pages 部署

### 方式 A：Wrangler 直接上传

第一次使用前先登录：

```powershell
npx wrangler login
```

检查登录状态：

```powershell
npx wrangler whoami
```

如果 API 通过 Cloudflare Tunnel 暴露为：

```text
https://your-api.trycloudflare.com
```

则构建前设置：

```powershell
$env:VITE_API_BASE_URL="https://your-api.trycloudflare.com"
$env:VITE_WS_URL="wss://your-api.trycloudflare.com/ws"
```

部署：

```powershell
npm run deploy:web:cloudflare
```

部署成功后，Wrangler 会返回一个类似这样的页面地址：

```text
https://feishu-ai-im-collab-assistant.pages.dev
```

当前线上项目可用以下命令复查：

```powershell
npm run doctor:deploy
```

如果本机代理导致 `pages.dev` 访问 TLS 握手失败，但 `wrangler pages deployment list` 能看到 Production 部署记录，说明 Cloudflare 侧已经发布，问题更可能在当前网络代理链路。

比赛现场建议直接运行：

```powershell
npm run demo:stack
```

该命令会启动本地 API、飞书事件桥接和 Cloudflare Tunnel，并输出可直接填入飞书后台的仪表盘 URL。

如果你不想每次临时 Tunnel 地址变化后都修改飞书后台，优先继续运行：

```powershell
npm run deploy:web-proxy:cloudflare
```

它会把 Cloudflare Pages 升级为“稳定仪表盘 + 边缘 API 代理”。飞书后台可直接填写固定 URL：

```text
https://feishu-ai-im-collab-assistant.pages.dev
```

后续临时 Tunnel 变化时，只需要重新运行这个命令更新 Pages 的 upstream。

如果你的 Cloudflare 账号已经注册 `workers.dev` 子域名，也可以运行独立 Worker Relay：

```powershell
npm run deploy:api-relay:cloudflare
```

它会部署一个 Cloudflare Worker Relay，并输出稳定的飞书仪表盘 URL。

如果已经有自有域名或 Cloudflare Tunnel token，则改用：

```powershell
npm run stable:stack
npm run doctor:stable-api
```

### 方式 B：Cloudflare Dashboard 连接 GitHub

在 Cloudflare 控制台进入：

```text
Workers & Pages -> Create -> Pages -> Connect to Git
```

填写：

| 字段 | 填写 |
| --- | --- |
| Project name | `feishu-ai-im-collab-assistant` |
| Production branch | `main` |
| Framework preset | `None` 或 `Vite` |
| Root directory | `/` |
| Build command | `npm ci && npm run build:web` |
| Build output directory | `apps/web/dist` |

环境变量：

| 变量 | 值 |
| --- | --- |
| `VITE_API_BASE_URL` | 你的 API HTTPS 地址 |
| `VITE_WS_URL` | 你的 API WebSocket 地址，例如 `wss://your-api-domain/ws` |

### 方式 C：用页面运行时配置连接 API

如果 Pages 已经部署，但当时没有设置 `VITE_API_BASE_URL`，也不用立刻重建。仪表盘支持在页面里填写 API / WS 地址，也支持通过 URL 参数临时覆盖：

```text
https://feishu-ai-im-collab-assistant.pages.dev/?api=https://your-api.trycloudflare.com
```

如果已经部署 Worker Relay，优先填 Relay 版本，之后更少改后台：

```text
https://feishu-ai-im-collab-assistant.pages.dev/?api=https://feishu-agent-api-relay.<your-subdomain>.workers.dev
```

如果已经部署 Pages 边缘代理，直接填这个即可：

```text
https://feishu-ai-im-collab-assistant.pages.dev
```

或：

```text
https://feishu-ai-im-collab-assistant.pages.dev/?api=https://your-api.trycloudflare.com&ws=wss://your-api.trycloudflare.com/ws
```

页面会把该配置保存到浏览器本地。适合比赛现场把 Cloudflare Tunnel 的临时 API 地址快速接到已部署页面。

## 4. 飞书开放平台怎么填写

### 4.1 添加应用能力

当前建议添加：

| 能力 | 状态 |
| --- | --- |
| 机器人 | 已添加，必须保留 |
| 网页应用 | 现在可以添加 |
| 工作台应用 | 可选，如果希望工作台里出现入口 |

暂时不需要：

- 工作台小组件
- 云文档小组件
- 多维表格插件
- 链接预览
- 消息快捷操作
- 原生集成应用

### 4.2 机器人配置

如何开始使用：

```text
在群聊中发送 /agent 或 @我，说出要整理的讨论，我会生成文档和PPT。
```

机器人自定义菜单：

```text
菜单状态：开启
展示形式：可切换菜单
默认展示：菜单
```

一级菜单最多 3 个。当前截图里已经有“开始整理 / 查看进度 / 取消任务”三个一级菜单，所以不会再出现“打开仪表盘”。推荐改成下面二选一。

方案 A：后台支持子菜单时。

| 一级菜单 | 二级菜单 | 响应动作 | 内容 / 链接 |
| --- | --- | --- | --- |
| 任务 | 开始整理 | 发送文字消息 | `/agent 请整理当前群聊讨论，生成需求文档、5页汇报PPT和3分钟讲稿。` |
| 任务 | 查看进度 | 发送文字消息 | `进度` |
| 任务 | 取消任务 | 发送文字消息 | `取消` |
| 仪表盘 | 无 | 跳转至指定链接 | `https://feishu-ai-im-collab-assistant.pages.dev` |
| 帮助 | 使用说明 | 发送文字消息 | `在群聊中发送 /agent 或 @我，说出要整理的讨论，我会生成文档和PPT。` |

方案 B：只用 3 个一级菜单时。

| 名称 | 响应动作 | 内容 |
| --- | --- | --- |
| 开始整理 | 发送文字消息 | `/agent 请整理当前群聊讨论，生成需求文档、5页汇报PPT和3分钟讲稿。` |
| 打开仪表盘 | 跳转至指定链接 | `https://feishu-ai-im-collab-assistant.pages.dev` |
| 取消任务 | 发送文字消息 | `取消` |

`查看进度` 可以不占菜单位，用户直接在群里发 `进度` 即可。菜单配置后必须创建版本并发布，客户端通常还会有几分钟缓存延迟。

`打开仪表盘` 的桌面端和移动端链接都先填同一个 Cloudflare Pages URL。若后端暂时通过 Cloudflare Tunnel 暴露，也可以填写带 `api` 参数的 URL：

```text
https://feishu-ai-im-collab-assistant.pages.dev/?api=https://your-api.trycloudflare.com
```

### 4.3 网页应用

添加“网页应用”后填写：

| 字段 | 填写 |
| --- | --- |
| 桌面端主页 | Cloudflare Pages 页面 URL |
| 移动端主页 | Cloudflare Pages 页面 URL |
| PC 端重定向地址 | 暂不填 |
| 移动端重定向地址 | 暂不填 |

当前仪表盘不做飞书免登，所以重定向 URL 先不需要。

### 4.4 工作台应用

如果添加“工作台应用”，主页也填：

```text
Cloudflare Pages 页面 URL
```

工作台入口用于演示“GUI 是仪表盘与辅助操作台”，不是主触发入口。主触发入口仍然是 IM 机器人。

### 4.5 事件与回调

事件配置：

```text
订阅方式：使用长连接接收事件
事件：接收消息 im.message.receive_v1
```

点击“验证”前先运行：

```powershell
npm run dev:lark-events
```

回调配置：

```text
暂不配置
```

加密策略：

```text
Encrypt Key：暂不填
Verification Token：平台生成即可，不写入代码
```

### 4.6 安全设置

重定向 URL：

```text
暂不填
```

IP 白名单：

```text
暂不开启
```

原因：

- 当前机器人长连接不需要 OAuth 重定向。
- 本地和 Cloudflare Tunnel 出口不固定，开 IP 白名单容易把自己拦掉。

## 5. 最小验收脚本

1. 本地启动 `npm run dev`。
2. 本地启动 `npm run dev:lark-events`。
3. 如需远程仪表盘访问本地 API，启动 `npm run tunnel:api:cloudflare`。
4. 打开 Cloudflare Pages 仪表盘 URL；如果使用 Tunnel，就打开带 `api` 参数的 URL。
5. 在飞书测试群发送：

```text
/agent 请整理刚才关于校园活动报名系统的讨论，生成需求文档、5页汇报PPT和3分钟讲稿。
```

6. 确认仪表盘出现 IM 来源任务。
7. 群里回复：

```text
确认
```

8. 打开最终回发的 Docs / Slides 链接，确认内容非空。
9. 发送：

```text
进度
```

确认机器人能回发当前任务状态。

## 6. 下一轮迭代

下一轮建议聚焦四件事：

1. 短期用 Pages 边缘代理固定飞书后台入口；中期将 Quick Tunnel 替换为 Named Tunnel + 自有域名，形成真正稳定 API URL。
2. 把“打开仪表盘”菜单接入飞书网页应用入口。
3. 做飞书网页应用免登，识别当前用户和当前群。
4. 把本地 Agent API 从临时 Tunnel 迁移到稳定云端运行环境，降低比赛现场网络风险。

稳定 API 的路线见：

```text

```
